# 20260425

> **Ghi nhớ**: PoW trong ZEN bây giờ hash toàn bộ canonical block `{#, ., :, >}` thay vì chỉ `key:nonce`. `field` không cần truyền vào `ZEN.pen({ pow })` — R[7] luôn là nonce register. Auto-mining diễn ra ngầm khi `put` nhận `opt.pow = {unit, difficulty}`. `sign` và `pow` là hai chính sách độc lập — có thể dùng riêng hoặc kết hợp.

---

Hôm nay tập trung vào hai mảng: (1) hoàn thiện kiến trúc PoW cho an toàn và đúng semantic, (2) dọn dẹp build pipeline.

## 1. Vấn đề bảo mật trong PoW cũ — `key:nonce` là không đủ

### 1.1 Trạng thái cũ

PoW trong ZEN trước đây kiểm tra:

```
SHA-256(key + ":" + nonce) phải bắt đầu bằng unit.repeat(difficulty)
```

Proof chỉ bind với `key` và `nonce`. Điều này có nghĩa là:
- Nonce tính cho key `"price"` có thể được tái sử dụng cho bất kỳ key nào khác tên `"price"` trên toàn mạng
- Giá trị `val` không được ràng buộc — kẻ tấn công có thể thay `val` mà giữ nguyên nonce
- Soul `#` không được ràng buộc — replay attack qua namespace khác là có thể

Đây là lỗ hổng nghiêm trọng nếu PoW được dùng như anti-spam hay như proof-of-cost.

### 1.2 Canonical block — giải pháp

Proof phải bind với toàn bộ khối dữ liệu, giống cách hash được thực hiện trong Bitcoin block header: tất cả các trường đều tham gia vào quá trình hash.

Canonical block trong ZEN là:

```js
const proofBlock = JSON.stringify({
  "#": soul,     // R[2] — ai owns namespace này
  ".": key,      // R[0] — key cụ thể
  ":": val,      // R[1] — giá trị được ghi
  ">": state,    // R[3] — HAM timestamp
})
// proof = proofBlock + ":" + nonce
```

Đây chính xác là những gì được ký khi `sign: true` — nên proof và signature bind với cùng một payload. Một nonce hợp lệ với block này sẽ không hợp lệ nếu bất kỳ trường nào thay đổi.

### 1.3 Triển khai — hai phía: mining và verification

**Mine** (`mineIfNeeded` trong `src/pen.js`):

```js
// Xây block TRƯỚC khi hash
var proofBlock = JSON.stringify({
  "#": ctx.put["#"] || "",
  ".": ctx.put["."] || "",
  ":": ctx.val,
  ">": ctx.put[">"] || 0,
})
// Hash loop: SHA-256(proofBlock + ":" + nonce)
// Nonce increment cho đến khi hash.startsWith(unit.repeat(difficulty))
// Kết quả lưu vào ctx.put["^"] = result.nonce
```

**Verify** (trong `runPredicate`):

```js
var nonce = regs[7] || ""  // R[7] — PoW nonce từ msg.put["^"]
var proofBlock = JSON.stringify({ "#": ..., ".": ..., ":": ctx.val, ">": ... })
var proof = proofBlock + ":" + nonce
// SHA-256(proof) phải startsWith unit.repeat(difficulty)
```

Peer nhận được nonce từ `msg.put["^"]` (được propagate qua mạng), rebuild canonical block từ dữ liệu thực tế, verify bằng cách hash lại. Nếu ai đó thay `val` thì block thay đổi → verification fail.

## 2. Auto-mining trong `put` — `opt.pow` là policy object

### 2.1 Trước đây: nonce được truyền thủ công

API cũ:

```js
// SAI — nonce cần biết trước mới truyền được
zen.get(soul).get(key).put(data, cb, { pow: "abc123" })
```

Pattern này sai ở chỗ: nonce phụ thuộc vào `val` và `soul` và `state` — những thứ chỉ biết tại thời điểm put. Người dùng không thể tính nonce trước khi gọi `put`.

### 2.2 Bây giờ: truyền policy object, ZEN tự mine

```js
// ĐÚNG — truyền policy, ZEN tự mine ngầm
zen.get(soul).get(key).put(data, cb, {
  authenticator: pair,             // optional — nếu policy có sign:true
  pow: { unit: "0", difficulty: 3 } // policy object, không phải nonce
})
```

`mineIfNeeded()` trong `penStage` nhận diện khi `opt.pow` là object (không phải string) và không có nonce trong `ctx.put["^"]`:

```js
function mineIfNeeded(cb) {
  var pow = sec.opt && sec.opt.pow
  if (pow && typeof pow === "object" && !ctx.put["^"]) {
    // Build canonical block, run SHA-256 loop
    runtime.hash(
      function(nonce) { return proofBlock + ":" + nonce },
      null,
      function(result) {
        ctx.put["^"] = result.nonce
        cb()
      },
      { pow: { unit: pow.unit || "0", difficulty: pow.difficulty ?? 3 },
        name: "SHA-256", encode: "hex" }
    )
  } else {
    cb()
  }
}
```

Peers re-propagating đã có `ctx.put["^"]` → `mineIfNeeded` là no-op với họ. Chỉ writer gốc mới mine.

### 2.3 Mặc định của unit và difficulty

- `unit` không truyền → `"0"` (hash phải bắt đầu bằng `"0"`)
- `difficulty` không truyền → `3` (hash phải bắt đầu bằng `"000"`)
- **Nếu `opt.pow` không được truyền khi `put` thì mining không diễn ra** — không có patch giá trị mặc định cho trường hợp này

## 3. `field` trong `ZEN.pen({ pow })` — không cần thiết

### 3.1 Câu hỏi từ dự án `akao`

Dự án `akao` đang sử dụng `ZEN.pen` như sau:

```js
this.pow = Statics?.system?.pow || { field: 7, difficulty: 3 }
// ...
ZEN.pen({
  key: { ... },
  sign: true,
  pow: this.pow  // → { field: 7, difficulty: 3 }
})
```

Comment trong code: `// Field 7 (R[7]) is reserved for Proof of Work nonce (["^"] in msg.put)`

Câu hỏi: **`field: 7` có thực sự cần thiết không?**

### 3.2 Trả lời: KHÔNG — `field` bị bỏ qua hoàn toàn

Trong bytecode compiler của `src/pen.js`:

```js
if (spec.pow) {
  var powfield = 7  // R[7] is ALWAYS reserved — không đọc spec.pow.field
  var powdiff  = (spec.pow.difficulty != null ? spec.pow.difficulty : 1) & 0xff
  var powunit  = spec.pow.unit ? String(spec.pow.unit).slice(0, 255) : ""
  // ...encode powfield, powdiff, powunit vào bytecode...
}
```

`spec.pow.field` không được đọc. `powfield` luôn là `7`. Truyền `field: 7` không có tác dụng gì — nó chỉ bị ignore.

**API đúng cho `akao`:**

```js
// Trước
this.pow = { field: 7, difficulty: 3 }

// Sau — bỏ field đi
this.pow = { difficulty: 3 }           // unit mặc định là "0"
this.pow = { unit: "0", difficulty: 3 } // tường minh hơn
```

R[7] là convention nội bộ của ZEN — không phải thứ developer cần nhớ hay truyền. Đó là lý do nó không nằm trong public API của `ZEN.pen`.

## 4. `sign` và `pow` — hai chính sách độc lập

Cả hai có thể được dùng riêng lẻ hoặc kết hợp trong một `ZEN.pen` spec:

```js
// Chỉ cần sign
ZEN.pen({ key: myExpr, sign: true })

// Chỉ cần PoW
ZEN.pen({ key: myExpr, pow: { unit: "0", difficulty: 3 } })

// Cả hai — phải vừa ký vừa mine
ZEN.pen({ key: myExpr, sign: true, pow: { unit: "0", difficulty: 3 } })
```

Khi `put` với cả hai policy:

```js
zen.get(soul).get(key).put(data, cb, {
  authenticator: pair,              // cần cho sign:true
  pow: { unit: "0", difficulty: 3 } // cần cho pow policy
})
```

Thứ tự xử lý trong `penStage`:
1. Nếu `sign: true` → sign/verify trước (ctx.val = verified plaintext)
2. `mineIfNeeded()` → mine canonical block với val đã được verify
3. `runPredicate()` → PEN VM kiểm tra bytecode
4. Nếu policy có `pow` → verify nonce
5. `applypolicy()` → nếu có `sign`/`cert` → final auth check

Sign trước, mine sau — đảm bảo canonical block chứa plaintext thực sự, không phải SEA blob.

## 5. Dọn dẹp build pipeline

### 5.1 `lib/pen.min.js` — xóa architectural drift

`lib/pen.min.js` là file tiền build (~12KB) không có `lib/pen.js` source tương ứng, không được import bởi bất kỳ file nào. Đây là tàn dư từ thời PEN còn là một lib plugin độc lập (`lib/pen.js`). Sau khi PEN được tích hợp vào `src/pen.js` và bundle vào `zen.js`, file cũ bị bỏ quên.

Tình trạng trước khi xóa:
- Import `../zen.js` (non-min) thay vì `../zen.min.js`
- Không có test nào import nó
- Không có `lib/pen.js` để build từ đó

→ **Xóa hoàn toàn.** PEN sống trong `src/pen.js` và được bundle vào `zen.js`.

### 5.2 `lib/uglify.js` — `rewriteImportsToMin()`

Trong quá trình `buildRelease`, khi minify `lib/*.js` → `lib/*.min.js`, các import paths cũng phải được rewrite từ `.js` → `.min.js`:

```js
function rewriteImportsToMin(code) {
  return code.replace(/(['"])(\.\.?\/[^'"]*?)\.js\1/g, function(match, q, path) {
    if (path.endsWith(".min")) return match
    return q + path + ".min.js" + q
  })
}
// Áp dụng: fs.writeFileSync(outPath, rewriteImportsToMin(result.code))
```

Kết quả: `npm run buildRelease` → 81 file minified, 0 non-min imports trong toàn bộ `lib/*.min.js`.
