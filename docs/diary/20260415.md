## 1. Multi-curve: ZEN vượt SEA

### Tư duy thay đổi

SEA được thiết kế cho một curve duy nhất — secp256k1. Mọi thứ hardcode xung quanh nó.

ZEN hôm nay đặt ra một nguyên tắc mới: **multi chains, multi curves**. Một key pair không phải chỉ là một cặp số. Nó là một danh tính có thể tồn tại trên nhiều chain, nhiều hệ thống mật mã, nhiều format khác nhau — tùy vào ngữ cảnh cần dùng.

```
ZEN.pair(null, { curve: 'secp256k1' })  → GUN/ZEN default
ZEN.pair(null, { curve: 'p256' })       → WebAuthn, TLS, iOS Secure Enclave
ZEN.pair(null, { format: 'evm' })       → Ethereum / EVM chains
ZEN.pair(null, { format: 'btc' })       → Bitcoin P2PKH
ZEN.pair(null, { curve: 'p256', format: 'evm' })  → P-256 identity → ETH address
```

Tất cả đều đi qua cùng một `pair()` API. Curve và format là opt-in. Default vẫn là secp256k1 — backward compatible hoàn toàn.

### Những gì được xây

**`src/p256.js`** — P-256 (secp256r1) curve engine:
- Pure JS, không dependency ngoài
- Cùng interface với secp256k1: `parsePub`, `parseScalar`, `pointMultiply`, `pointToPub`, `shaBytes`, `base62`
- Dùng WebCrypto `P-256` cho sign/verify (native browser/Node)

**`src/ripemd160.js`** — Pure JS RIPEMD-160:
- 101 dòng, không dependency
- Cần cho Bitcoin P2PKH address: `RIPEMD160(SHA256(pubkey))`

**`src/format.js`** — Format converters:
- `evm(priv, pub, epub)` → `{ pub: '0x<checksum>', priv: '0x<hex>', epub: '0x04<x><y>' }`
  - EIP-55 checksum: `keccak256(hex_address)` → uppercase/lowercase theo bit
- `btc(priv, pub, epub)` → `{ pub: '1<base58>', priv: '<WIF>', epub: '0x02/03<x>' }`
  - P2PKH: `Base58Check(0x00 + RIPEMD160(SHA256(compressed_pub)))`
  - WIF: `Base58Check(0x80 + priv + 0x01)`

**`src/curves/curves.js`** — Curve dispatcher:
- `crv('secp256k1')` → secp256k1 engine
- `crv('p256')` / `crv('secp256r1')` → p256 engine
- Tất cả đi qua đây — một điểm duy nhất để thêm curve mới

**`src/pair.js`** — Rewrite để multi-curve:
- Nhận `opt.curve` và `opt.format`
- Deterministic từ seed trên cả hai curves
- `opt.priv` restore từ private key

**`src/sign.js`, `verify.js`, `secret.js`, `encrypt.js`, `decrypt.js`** — Tất cả multi-curve:
- `sign()` detect curve từ `pairLike.curve`, dispatch đúng engine
- Envelope có `c: 'p256'` nếu dùng P-256, không có `c` nếu secp256k1 — backward compat
- `verify()` đọc `c` field để biết dùng curve nào
- `secret()` ECDH trên cả hai curves

### Test coverage

**`test/zen/multicurve.js`** — 288 dòng, bao phủ:
- secp256k1 backward compat (pub format, additive derivation)
- P-256: pair, seed determinism, restore from priv, cross-curve khác nhau
- EVM format: checksummed address, 0x+64hex priv, uncompressed epub, EIP-55
- BTC format: P2PKH starts with `1`, WIF `K/L`, compressed epub `0x02/03`
- Callback API với formats
- sign/verify multi-curve roundtrip
- `c: 'p256'` trong envelope, không có `c` với secp256k1
- Cross-curve verification fail (đúng hành vi)
- ECDH secret multi-curve

**`test/zen/crypto.js`** — 499 dòng, bao phủ mọi primitive riêng lẻ.

### RIPEMD-160 test vectors

Bitcoin P2PKH address từ scalar = 1 (điểm G): `1BgGZ9tcN4rm9KBzDn7KprQz87SZ26SAMH` — đây là địa chỉ Bitcoin nổi tiếng nhất trong cryptographic test vectors. ZEN tạo đúng format, dù không test exact address vì không inject scalar trực tiếp vào pair().

### Tại sao P-256 quan trọng

- **WebAuthn/Passkey**: iOS Secure Enclave và Android Strongbox chỉ hỗ trợ P-256
- **TLS/HTTPS**: certificate authorities dùng P-256
- **Nhiều HSM và smart card**: P-256 native, secp256k1 không có
- **Cross-chain identity**: cùng một user có thể có identity trên EVM (secp256k1) và trên hệ thống WebAuthn (P-256), ZEN bridge được cả hai

### Tại sao multi chains là hướng đúng

ZEN không phải là "GUN SEA với secp256k1". ZEN là lớp identity và crypto cho bất kỳ chain nào, bất kỳ curve nào. Một user trong ZEN có thể:

- Sign message cho GUN graph (secp256k1, base62 format)
- Derive Ethereum wallet (secp256k1, EVM format)
- Derive Bitcoin address (secp256k1, BTC format)
- Tạo WebAuthn-compatible key (P-256)
- Certify quyền write cho người khác (certify API, multi-curve)

Tất cả từ cùng một `ZEN.pair()`.

## 2. Port `sea.certify` → `zen.certify`

### Tại sao cần thiết

`certify` là một trong những API quan trọng nhất của GUN SEA — nó cho phép một authority cấp certificate để người khác có quyền write vào graph của họ mà không cần share private key.

```
ZEN.certify(certificants, policy, authority)
→ JSON string cert  (verify được bằng ZEN.verify)
```

Trước đó `src/index.js` đã có `ZEN_SKIP = { certify: 1 }` — nghĩa là slot đã được giữ chỗ nhưng chưa có implementation. Hôm nay slot đó đã được lấp đầy.

### Những gì đã làm

**`src/certify.js`** — ZEN port:

- Tên biến ngắn theo ZEN style: `certs`, `pol`, `auth`, `blk`, `r`, `w`, `rb`, `wb`
- Dùng `sign()` từ `./sign.js` — tức là certify tự động kế thừa multi-curve (secp256k1 + p256)
- Không có prefix `SEA` trong output — output là plain `JSON.stringify(cert)`
- `opt.raw` trả về raw cert object thay vì string

**`src/index.js`**:

- `import certify from './certify.js'`
- Thêm vào `SECP256K1` object
- `ZEN_SKIP = {}` — slot đã được bỏ
- Static method `ZEN.certify()` + instance method `zen.certify()`

**`test/zen/certify.js`** — 29 tests:

- Basic write policy (certify + verify)
- Wildcard certificants (`'*'`, array chứa `'*'`)
- Array certificants (multi-pub, single-element unwrap, array of objects)
- Object certificant với `.pub`
- Policy forms: string, RAD/LEX object, array, `policy.read + policy.write`, `policy.read` only
- Expiry (`opt.expiry` number và string)
- Block lists (`opt.block`, `.blacklist`, `.ban`, `.read`, `.write`, soul ref)
- `opt.raw`
- Callback
- Error cases (null certs, empty object, no policy)
- Multi-curve authority: p256 authority signs cert, curve marker `c: 'p256'` trong envelope, verify thành công

**`test/zen/instance.js`**:

Cần update 1 line: test cũ expect `typeof zen.certify === 'undefined'` (do ZEN_SKIP) — đổi thành `'function'`.

### Điều quan trọng về wire format

`sign()` với `{ raw: 1 }` trả về object `{ m: parsedObject, s: '...' }` — không phải string.

Vì vậy khi đọc lại cert data trong test, phải handle cả 2 trường hợp:

```js
const data = typeof cert.m === 'string' ? JSON.parse(cert.m) : cert.m;
```

Đây là một chi tiết quan trọng về cách `finalize()` hoạt động trong secp256k1: `opt.raw` trả về object thay vì stringify nó.

## 3. Dọn emailjs

Mark đã nhúng `emailjs` vào 3 chỗ:

| File | Code bị xóa |
|------|-------------|
| `lib/axe.js` | `import __email` + toàn bộ "UNIVERSAL NOTIFICATION MODULE" IIFE (~30 dòng) — email tagging system gửi notification qua SMTP |
| `lib/crashed.js` | `import __email` + `function email()` + guard `process.env.EMAIL` — tự động email V8 crash log cho dev |
| `lib/service.js` | `cmd.https` — cung cấp SSL cert qua Let's Encrypt bằng cách nhận `msg.email` từ peer |

Cả 3 đều là opt-in developer tools, không phải core protocol. Sau khi `npm uninstall emailjs` và xóa `email.js`, không còn lý do giữ lại chúng.

Sau khi dọn: `grep -rn "email" lib/axe.js lib/crashed.js lib/service.js` — không có kết quả.

## 4. Dọn text-encoding

`lib/text-encoding/` là một vendored copy của polyfill `TextEncoder`/`TextDecoder` từ năm ~2019, thời điểm chúng chưa phải native global.

Năm 2026, `TextEncoder` và `TextDecoder` là native trong:
- Node.js (từ v11, 2018)
- Tất cả browser hiện đại (từ ~2018)
- React Native (từ 0.60+)

Những chỗ đã xóa:

| File | Thay đổi |
|------|----------|
| `lib/mobile.js` | Bỏ `import { TextEncoder, TextDecoder } from 'text-encoding'` và 2 dòng `global.TextEncoder/TextDecoder =` |
| `examples/react-native/shim.js` | Bỏ `import { TextEncoder, TextDecoder } from 'text-encoding'` |
| `examples/react-native/package.json` | Bỏ `"text-encoding": "^0.7.0"` khỏi dependencies |
| `lib/text-encoding/` | Xóa toàn bộ folder |

Sau khi dọn: `grep -rn "text-encoding"` trong zen (trừ package-lock) — không có kết quả.

## Tổng kết

Hôm nay ZEN vượt SEA về crypto primitives:

- **multi-curve** là thay đổi lớn nhất — ZEN không còn bị gán với một elliptic curve hay một chain duy nhất. Tư duy mới: **multi chains, multi curves**. Một identity có thể sống trên GUN graph, Ethereum, Bitcoin, WebAuthn — tất cả từ cùng một `pair()`.
- **certify** lấp đầy API slot quan trọng cuối cùng — `ZEN_SKIP = { certify: 1 }` đã được xóa. ZEN surface giờ hoàn chỉnh hơn SEA.
- **emailjs + text-encoding** là hai thứ không thuộc về ZEN ngay từ đầu — dọn chúng ra là làm cho dependency graph trở nên trung thực hơn.

Về crypto primitives, SEA không còn gì để ZEN học thêm. Mọi thứ đã được port, nhiều thứ đã vượt qua.

Một codebase tốt không chỉ là code được thêm vào.

Nó cũng là code được xóa đúng lúc — và code được thiết kế lại để sống được ở nhiều nơi cùng một lúc.
