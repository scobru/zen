## 1. CLI crash trên POSIX dash — `shift || true`

**Commit:** `33f70fe` — `fix: CLI shift error + replace browser WS peer scan with fetch`

### Vấn đề

`script/zen.sh` có dòng:

```sh
shift || true
```

Trên `bash` hoặc `zsh` với `set -eu`, expression này hoạt động bình thường — khi `$# = 0`, `shift` fail nhưng `|| true` suppress lỗi. Tuy nhiên trên POSIX `dash` (shell mặc định trên Debian/Ubuntu, thường dùng làm `/bin/sh`), `shift || true` trong `set -eu` context vẫn propagate lỗi ngay cả khi có `|| true` — POSIX behavior khác với bash extension.

### Giải pháp

```sh
# Trước
shift || true

# Sau
if [ "$#" -ge 1 ]; then shift; fi
```

Explicit guard là portable 100% trên mọi POSIX shell.

## 2. Browser console spam — bscan() WebSocket flood

**Commit:** `33f70fe` (cùng commit)

### Vấn đề

Khi browser load zen, hàm `bscan()` trong `src/axe.js` tạo blind WebSocket connections tới `peer0.akao.io:8420`, `peer1.akao.io:8420`, ..., `peer100.akao.io:8420` để khám phá peers. Với mỗi host không tồn tại, browser log một `ERR_CONNECTION_REFUSED` hoặc DNS error vào DevTools console.

**Quan trọng:** try/catch **không** suppress được lỗi này — browser log lỗi WebSocket ở network layer trước khi bất kỳ JS handler nào chạy. Không có cách nào dùng JS để im lặng lỗi đó.

### Giải pháp

Xóa hoàn toàn `bscan()`. Bước bootstrap 5 thay bằng:

```js
fetch(origin + "/status")
  .then(r => r.json())
  .then(s => { /* s.peers là danh sách relay peers */ })
  .catch(() => {}); // fail silently, không có console error
```

`/status` endpoint đã có CORS header, trả về JSON với danh sách `peers`. `fetch` fail mà không log console khi `.catch(() => {})` được đặt.

## 4. Memory leaks — on.js, websocket.js, axe.js

**Commit:** `bc17eef` — `fix: plug memory leaks in on.js, websocket.js, axe.js`

### 4a. `src/on.js` — subscription accumulation

`trackSub()` không tự xóa subscription khỏi `eas.subs[]` khi teardown. Trên relay chạy dài ngày, mảng này tích lũy không giới hạn. Fix: `trackSub()` giờ self-splice khỏi `eas.subs[]`. `chain.off()` clear toàn bộ: once timers, any listeners, subs, jam queue.

### 4b. `src/websocket.js` — timer accumulation

`reconnect()` không null `peer.defer` sau `clearTimeout`. Inner `setTimeout` không re-check `canReconnect()`. Kết quả: nhiều reconnect timers có thể fire đồng thời sau khi connection đã alive. Fix: `peer.defer` luôn được null sau `clearTimeout`; inner `setTimeout` re-check trước khi chạy và lưu handle vào `peer.defer`; `onopen` clear `peer.defer` ngay lập tức.

### 4c. `src/axe.js` — dead peers trong route maps

Bye handler không clear timers và không xóa peer khỏi soul route Maps. Fix: `clearTimeout(peer.to)` + `clearTimeout(peer.defer)`. Null `peer.next`/`peer.put`. Iterate `peer.sub`, xóa peer khỏi mỗi soul's `ref.route` Map, rồi null `peer.sub`.

## 6. WebRTC Browser-to-Browser P2P — 4 bugs, 1 feature

Đây là phần chính của ngày. WebRTC đã được implement từ trước nhưng DataChannel không bao giờ thực sự mở được. Sau quá trình debug systematically từng layer, tìm ra 4 bugs riêng biệt:

---

### Bug 1 — Relay không gửi `bpids` trong PEX

**Commit:** `530bc0a` — `relay sends PEX with bpids on new inbound connection for WebRTC auto-discovery`

#### Vấn đề

`lib/webrtc.js` có `mesh.hear["pex"]` hook, khi nhận `msg.bpids` (mảng browser peer IDs) sẽ gọi `opt.rtc.connect(pid)` để initiate WebRTC. Nhưng relay trong `src/axe.js` **không bao giờ** populate `bpids` khi gửi PEX — browsers không biết peer ID của nhau → WebRTC không bao giờ được khởi tạo.

#### Giải pháp

Trong `src/axe.js`, relay-side `mesh.hear["?"]` override, khi browser mới kết nối:

1. Gửi `{ dam:"pex", peers:[relay URLs], bpids:[existing browser pids] }` cho browser mới
2. Broadcast `{ dam:"pex", bpids:[new browser pid] }` cho tất cả browsers đang online

Kết hợp với rule "lower pid initiates" trong `webrtc.js` (`if (opt.pid >= pid) return`), tránh double-offer race.

---

### Bug 2 — `drop()` bỏ qua open DataChannels

**Commit:** `4bdb38a` — `fix WebRTC drop(): handle open DataChannels + readyState guard with relay fallback`

#### Vấn đề

Lifecycle của WebRTC peer có 2 giai đoạn:
- **Pending**: `pcs[pid]` = RTCPeerConnection đang handshake
- **Wired**: `dc.onopen` → xóa `pcs[pid]`, gọi `mesh.hi(pc)` → peer vào `opt.peers[pid]`

`drop()` cũ chỉ check `pcs[pid]`. Khi DataChannel đã open và `pcs[pid]` đã bị xóa, `drop()` return sớm mà không dọn `opt.peers[pid]`. Dead DataChannels tích lũy.

#### Giải pháp

- `drop()` check cả `pcs[pid]` (pending) lẫn `opt.peers[pid]` (wired), gọi `mesh.bye()` cho wired peer
- `pc.say()` check `dc.readyState === "open"` trước khi gửi; nếu fail → gọi `drop(pid)` → traffic tự động fallback về relay WebSocket
- `dc.onerror` gọi `drop(pid)` (trước đây empty)

---

### Bug 3 — `mesh.hear["rtc"]` không được đăng ký

**Commit:** `b5ec256` — `fix: relay must register mesh.hear["rtc"] = rtcway`

#### Vấn đề

**Kiến trúc quan trọng của `src/mesh.js`:** Hàm `hear.one()` xử lý mọi message có `dam` field bằng cách dispatch qua `mesh.hear[dam]` và **luôn return sớm** — `mesh.way` không bao giờ được gọi cho `dam` messages.

`axe.js` đã wire `rtcway` vào `mesh.way` — nghĩa là WebRTC signaling (offer/answer/ICE) không bao giờ được relay route.

#### Giải pháp

```js
mesh.hear["rtc"] = rtcway;
```

Thêm dòng này ngay sau definition của `rtcway` trong `src/axe.js`. Sau fix này, OFFER và ANSWER SDP được exchange thành công.

### Bug 4 — ICE candidates serialize thành `[object Object]`

**Commit:** `2d3c5f2` — `fix: toJSON() on RTCIceCandidate before sig() — plain object for serialisation`

#### Vấn đề

Sau khi fix Bug 3, Playwright WebSocket spy cho thấy wire payload chứa `"candidate":[object Object]` — JSON không hợp lệ tại nguồn. ICE candidates luôn là `[]` empty array.

Root cause: `RTCIceCandidate` (Chrome Web API) store `candidate`, `sdpMid`, `sdpMLineIndex`, `usernameFragment` là **prototype getters**, KHÔNG phải own enumerable properties. `JSON.stringify(iceCandidate)` hoạt động khi gọi trực tiếp (invoke `toJSON()`), nhưng khi object được nhúng vào GUN message và serialize qua ZEN's JSON pipeline, prototype getters không được access → object coerce thành string `[object Object]`.

#### Giải pháp

```js
// Trước
sig({ candidate: e.candidate, pid: opt.pid, to: pid });

// Sau
sig({ candidate: e.candidate.toJSON(), pid: opt.pid, to: pid });
```

Gọi `.toJSON()` tại nguồn → plain serialisable object → pipeline xử lý đúng.

### Feature — MAX_RTC_PEERS enforcement

**Commit:** `a28e1dd` — `feat: enforce opt.rtc.max browser peer limit in WebRTC`

#### Vấn đề

WebRTC là O(n²): n browsers → n(n-1)/2 DataChannels. Tại n=100: 4,950 simultaneous RTCPeerConnections. Chrome crash trước con số đó. `opt.rtc.max = 55` đã khai báo từ trước (WebTorrent convention, dưới Chrome hard limit ~256) nhưng **chưa bao giờ được enforce**.

#### Implementation

```js
// Tracking
const dcPeers = {};       // confirmed-open DataChannels
// pcs = {} đã có từ trước (pending connections)

const rtcCount = () => Object.keys(pcs).length + Object.keys(dcPeers).length;

// dc.onopen
dcPeers[pid] = true;

// drop()
delete dcPeers[pid];

// Guard — initiator side
if (rtcCount() >= opt.rtc.max) return;

// Guard — responder side
if (!pcs[pid] && rtcCount() >= opt.rtc.max) return;
// (!pcs[pid]) vì nếu đang pending thì không phải "mới" → không block

// Random peer selection trong PEX handler
// Fisher-Yates shuffle msg.bpids trước khi iterate
// → khi full, pick random peers thay vì luôn alphabetically first
```

## Bài học kỹ thuật hôm nay

**1. WebSocket errors không thể suppress bằng try/catch**
Browser log lỗi WebSocket ở network layer trước khi bất kỳ JS nào chạy. Giải pháp duy nhất: đừng tạo kết nối thất bại. `fetch` với `.catch(() => {})` hoạt động.

**2. `shift || true` không portable trên POSIX dash**
POSIX `dash` xử lý `set -eu` khác `bash`. Luôn dùng explicit conditional cho portable shell scripts.

**3. `mesh.hear[dam]` vs `mesh.way` — architectural constraint**
`hear.one()` trong `mesh.js` dispatch `dam` messages qua `mesh.hear[X]` và **luôn return sớm**. Bất kỳ handler nào cần nhận `dam` messages **phải** được đăng ký vào `mesh.hear`, không phải `mesh.way`.

**4. Web API prototype getters không serialize khi nhúng vào object**
`RTCIceCandidate`, `RTCSessionDescription`, và nhiều Web API khác dùng prototype getters. `JSON.stringify(api_obj)` hoạt động trực tiếp (gọi `toJSON()`), nhưng `JSON.stringify({ nested: api_obj })` không gọi `toJSON()` của nested object — coerce thành `[object Object]`. Luôn gọi `.toJSON()` tại nguồn trước khi nhúng vào message.

**5. Debug WebRTC requires layer-by-layer elimination**
WebRTC có nhiều layers: signaling routing → SDP exchange → ICE gathering → ICE connectivity → DataChannel open. Khi debug, phải confirm từng layer từ dưới lên bằng công cụ thực tế (Playwright WS spy, DevTools ICE stats). Không thể đoán layer nào fail.
