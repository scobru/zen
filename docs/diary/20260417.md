## Buổi sáng: Windows, đường dẫn, và format toàn repo

Đây không phải kiểu ngày làm feature mới rồi chụp ảnh đẹp. Đây là kiểu ngày phải bò qua từng lớp bug để đưa repo về trạng thái ổn định sau một loạt thay đổi lớn.

### 1. Windows không phải môi trường ngoại lệ nữa

Có những ngày bug nằm ở business logic. Hôm nay bug nằm ở những thứ nhỏ, dai, và rất tốn sức: đường dẫn, shell behavior, môi trường runtime, và các assumption ngầm mà trước đó Unix chịu được còn Windows thì không.

Nó tạo ra chuỗi triệu chứng: test treo ở những chỗ bất đồng bộ, command chạy được ở một môi trường nhưng lệch ở môi trường khác, file/path đổi vị trí là build đổ dây chuyền.

Phần lớn thời gian là để gỡ từng assumption cũ ra khỏi repo, để project cư xử ổn định hơn với Windows thay vì chỉ "may mắn chạy được".

### 2. `src/zen/*` được dời ra `src/*`

Thay đổi cấu trúc đúng đắn — nhưng nó bóc ngay những import cũ còn sót. Build hỏng không phải vì bundler xấu, mà vì trong code vẫn còn đường dẫn kiểu cũ:

```js
// cũ — không còn tồn tại
import shim from './zen/shim.js'

// mới — layout thật
import shim from './shim.js'
```

Sau khi dọn hết tham chiếu cũ, `npm run buildZEN` quay về bình thường. `src/*` là layout thật, không phải một lớp rename nửa vời.

### 3. Format toàn repo

Script `format` được đổi sang `prettier --write .` với `.prettierignore` được cấu hình có chủ đích — loại trừ runtime artifacts, data dirs, minified outputs. `npm run format` giờ là command dùng được thật, không phải nút trang trí trong `package.json`.

Kết thúc buổi sáng: `npm test` xanh. Ba thay đổi rất khác nhau đã cùng đứng vững.

## Buổi chiều: Zig, WASM, và bài học về việc đo trước khi tối ưu

Đây là session thú vị nhất: ngày đầu tiên ZEN có một pipeline WASM thứ hai bên cạnh PEN.

Xuất phát từ một quan sát đúng: toàn bộ crypto — SHA-256, Keccak-256, RIPEMD-160, HMAC, base62, ECC point multiply — đang chạy bằng BigInt thuần JavaScript. Với `deterministicK` (RFC 6979) và `pointMultiply` (Weierstrass scalar mul), điều này trở nên chậm rõ ràng dưới tải test.

Mục tiêu ban đầu: port toàn bộ sang Zig, compile ra WASM.

### Những gì đã được xây dựng

9 module Zig:

| Module | Nội dung |
|--------|---------|
| `src/sha256.zig` | Streaming SHA-256, FIPS 180-4 |
| `src/keccak256.zig` | Keccak-256, 25 lane u64 |
| `src/ripemd160.zig` | RIPEMD-160 thuần |
| `src/hmac_sha256.zig` | HMAC streaming, zero-alloc |
| `src/base62.zig` | Encode/decode u256/u512 |
| `src/curves/utils.zig` | Weierstrass: modPow, pointAdd, pointMultiply, RFC 6979 |
| `src/curves/secp256k1.zig` | secp256k1 params |
| `src/curves/p256.zig` | P-256 params |
| `src/crypto.zig` | WASM entry: bump allocator 1MB, 20 exports |

Tất cả compile thành `crypto.wasm` — **66138 bytes**. Bridge: `src/crypto.js`. Pipeline: `buildPEN → buildCrypto → builder/zen.js → minify`.

### Cú ngã và bài học thực tế

Sau khi wire HMAC, deterministicK, pointMultiply sang WASM — kết quả test gây sốc:

**Test suite chậm hơn 7–12 lần.** `sign()` timeout ở mocha default 2000ms.

| Hot path | Cũ | WASM | Kết quả |
|----------|-----|------|---------|
| `hmacSha256` | WebCrypto `subtle.sign("HMAC")` — hardware SHA-NI | WASM software SHA-256 | **thua 7×** |
| `deterministicK` | WebCrypto HMAC loop (RFC 6979) | WASM HMAC loop | **thua 7×** |
| `pointMultiply` | V8 BigInt double-and-add | WASM u256×u256 emulated | **thua 8–12×** |

Vấn đề cốt lõi: WebCrypto dùng hardware SHA-NI. V8 BigInt là native C++ với hardware multiply. WASM32 không có wide multiply — phải emulate bằng chuỗi 32-bit ops.

### Quyết định

**Giữ WASM:** keccak256 (25-lane u64, không có WebCrypto equivalent), ripemd160 (pure compute), base62 (nhanh hơn BigInt cho encoding).

**Revert về native:** pointMultiply → V8 BigInt, hmacSha256 + deterministicK → WebCrypto hardware.

### Kết quả

| Test | Trước WASM | Sau WASM (sai) | Sau revert |
|------|-----------|---------------|-----------|
| sign + encrypt omit prefix | ~300ms | 2191ms (timeout) | **208ms** |
| signs deterministically | ~560ms | 6074ms | **566ms** |
| pair → encrypt → sign → verify | ~400ms | 3850ms | **406ms** |

### Bài học

> Điều quan trọng không phải là JS hay Zig — mà là cái gì nhanh hơn thì dùng cái đó.

1. **Đo trước khi tối ưu.** WASM không tự động nhanh hơn JS.
2. **WASM thắng khi:** pure compute không có WebCrypto equivalent (Keccak-256, RIPEMD-160).
3. **WASM thua khi:** platform đã có hardware acceleration (SHA-256, AES, HMAC).
4. **WASM32 thua BigInt cho scalar multiply** — V8 BigInt là native C++, WASM32 phải emulate wide arithmetic.

## Buổi tối: Nghiên cứu kiến trúc, benchmark harness, và tối ưu đầu tiên từ dữ liệu thực

Sau WASM, câu hỏi là: bước tiếp theo là gì? Câu trả lời: **đọc kỹ kiến trúc từ dưới lên, đo ở mức micro, chỉ tối ưu sau khi có số thực.** Không làm đại trà như buổi chiều.

### Nghiên cứu kiến trúc 9 lớp

| Layer | Module | Vai trò |
|-------|--------|--------|
| 1 | `src/shim.js` | `setTimeout.turn` (MessageChannel), `setTimeout.each`, `String.random`, `String.hash` DJB2 |
| 2 | `src/dup.js` | Message dedup — gọi trên **mọi** message |
| 3 | `src/state.js` | CRDT vector clock — `State()` monotonic timestamp, `State.ify` merge |
| 4 | `src/valid.js` | Value validation — chỉ cho phép `null/string/boolean/number/soul ref` |
| 5 | `src/onto.js` | Doubly-linked list event emitter — O(1) subscribe, O(n) traverse |
| 6 | `src/root.js` | `universe(msg)` — pipeline trung tâm: dedup → HAM → put/get/ack → forward |
| 7 | `src/mesh.js` | P2P networking: `hear(raw, peer)`, JSON parse, batch 9 items/turn |
| 8 | `lib/radix.js` | Compressed prefix tree — insert/lookup character-by-character |
| 9 | `lib/radisk.js` | Batch storage: `until=250ms`, `batch=10K ops`, wraps Radix |

Hot path thực sự trên mỗi message:

```
JSON.parse(raw)
  → dup.check(msg["#"])     // String.hash DJB2 + Map lookup
  → dup.track(msg["#"])     // s.set(id, {was: now})
  → HAM(val, key, soul)     // arithmetic comparison per key
  → put → graph merge       // Object.keys(put) iteration
```

### Xây dựng micro-benchmark harness

`test/bench/` — framework zero-dependency:

- `harness.js` — `suite()` + `bench()` + `run({warmup, iters})`, async-native, `performance.now()`, JSON output
- `01_hash.js` — SHA-256, sha256.js, keccak256 WASM, ripemd160 WASM, ZEN.hash, String.hash DJB2
- `02_json.js` — JSON.parse/stringify, parseAsync shim, YSON chunk parser
- `03_dup.js` — dup.check/track, Map baseline, Set baseline, String.hash DJB2
- `04_radix.js` — Radix insert/lookup/scan vs native Map
- `05_ham.js` — State(), HAM comparisons, Object.keys cost, ZEN put() e2e
- `06_crypto_sign.js` — ZEN.pair/sign/verify/encrypt, secp256k1 primitives
- `07_base62.js` — base62 WASM encode/decode vs base64

```bash
npm run bench          # chạy tất cả
npm run bench:hash     # chỉ hash
npm run bench:dup      # chỉ dedup
```

### Tối ưu đầu tiên từ dữ liệu: `dup.s` plain object → `Map`

`dup.check()` + `dup.track()` được gọi trên **mọi** message đến. Với random-looking string keys (9-char IDs, soul hashes), V8 không optimize object property lookup tốt bằng `Map`.

Thay đổi trong `src/dup.js`:

```js
// cũ
var dup = { s: {} };
if (!s[id]) { ... }
var it = s[id] || (s[id] = {});

// mới
var dup = { s: new Map() };
if (!s.has(id)) { ... }
var it = s.get(id); if (!it) { it = {}; s.set(id, it); }
```

8 chỗ trong `src/mesh.js` cũng phải cập nhật.

**Kết quả sau tối ưu:**

| Operation | Trước | Sau | Speedup |
|-----------|-------|-----|---------|
| check missing | 2.57M | **4.54M** | **+77%** |
| track new id | 956K | **2.04M** | **+113%** |
| full pipeline | 815K | **1.13M** | **+38%** |

## Nhìn lại

Ba thứ đã thay đổi hôm nay theo nghĩa thật:

**1. Mindset về WASM**: Không phải "port sang Zig là nhanh hơn". Mà là "đo trước, chỉ port khi số thực cho thấy WASM thắng". Keccak + RIPEMD-160 thắng. SHA-NI + V8 BigInt thắng platform. Nguyên tắc này sẽ định hướng mọi quyết định tối ưu tiếp theo.

**2. Benchmark infrastructure**: Giờ có `test/bench/` với 7 suites. Mỗi quyết định tối ưu trong tương lai bắt đầu từ đây — không phải từ trực giác.

**3. Tối ưu đầu tiên từ dữ liệu**: `dup.s` Map — không cần viết một dòng Zig, nhưng thực, có thể đo, có mặt trong production ngay hôm nay.

Đây là kiểu ngày mà hệ thống trưởng thành hơn một cách thật sự.
