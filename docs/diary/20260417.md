## Ngày của Windows, đường dẫn cũ, và một lần format toàn repo

Hôm nay không phải kiểu ngày làm feature mới rồi chụp ảnh đẹp.

Hôm nay là kiểu ngày phải bò qua từng lớp bug để đưa repo về trạng thái ổn định sau một loạt thay đổi lớn.

Có 3 việc chính chồng lên nhau:

1. sửa các vấn đề tương thích, đặc biệt là trên Windows
2. di chuyển toàn bộ `src/zen/*` ra `src/*`
3. chạy format code toàn diện bằng Prettier mà không làm vỡ build hay test

Điểm tốt là cuối ngày, mọi thứ đều quy về một kết quả rõ ràng:

- build chạy lại được
- test xanh hết
- codebase gọn hơn lúc bắt đầu

---

## 1. Hôm nay thực sự là ngày fix bug để sống sót trên Windows

Có những ngày bug nằm ở business logic. Hôm nay bug nằm ở những thứ nhỏ, dai, và rất tốn sức: đường dẫn, shell behavior, môi trường runtime, và các assumption ngầm mà trước đó Unix chịu được còn Windows thì không.

Loại bug này mệt ở chỗ nó không tạo ra một lỗi duy nhất.

Nó tạo ra chuỗi triệu chứng:

- test treo ở những chỗ bất đồng bộ
- command chạy được ở một môi trường nhưng lệch ở môi trường khác
- file/path đổi vị trí là build đổ dây chuyền
- formatter hoặc tooling đụng vào code cũ thì lộ ra thêm lỗi parse legacy

Phần lớn thời gian hôm nay không phải để viết tính năng mới.

Phần lớn thời gian là để gỡ từng assumption cũ ra khỏi repo, để project cư xử ổn định hơn với Windows thay vì chỉ “may mắn chạy được”.

Đó là loại tiến triển ít hào nhoáng nhưng rất thật.

## 2. `src/zen/*` được dời ra `src/*` và build vỡ ngay lập tức

Đây là thay đổi cấu trúc đúng đắn, nhưng nó cũng bóc ngay những chỗ import cũ còn sót lại.

Sau khi flatten source tree, build không hỏng vì bundler “xấu”. Build hỏng vì trong code vẫn còn nhiều đường dẫn kiểu cũ trỏ đến:

- `./zen/shim.js`
- `../zen/shim.js`

Khi `src/zen/*` không còn nữa, những import này trở thành `ENOENT` rất nhanh và rất rõ.

Việc cần làm không phải là vá build script theo kiểu vòng vo. Việc đúng là sửa tận gốc các import stale trong source để chúng phản ánh đúng layout mới:

```js
// cũ
import shim from './zen/shim.js'

// mới
import shim from './shim.js'
```

và tương tự với các file trong `src/curves/` dùng `../shim.js`.

Sau khi dọn hết các tham chiếu cũ, `npm run build` quay lại trạng thái hoạt động bình thường.

Điểm quan trọng ở đây là: repo bây giờ không còn sống dựa trên layout cũ nữa. `src/*` đã trở thành layout thật, không phải chỉ là một lớp rename nửa vời.

## 3. Format toàn repo: không còn chỉ là `src/*.js`

Một thay đổi khác hôm nay là đưa formatting từ mức tượng trưng sang mức thực dụng.

Trước đó script format chỉ nhắm vào một tập rất hẹp. Mục tiêu hôm nay là rõ ràng hơn: nếu đã format thì format **toàn codebase**, không phải chỉ vài file trong `src/`.

Script `format` được đổi sang chạy Prettier trên toàn repo:

```json
"format": "prettier --write ."
```

Nhưng chạy Prettier toàn repo không đơn giản là đổi một dòng script rồi xong.

Ngay lập tức xuất hiện 2 lớp vấn đề:

### 3.1. `.prettierignore` cũ chặn gần như mọi thứ

Nó phải được thay bằng danh sách ignore có chủ đích hơn:

- thư mục runtime / generated
- data directories
- minified outputs
- artifact build như `zen.js`, `zen.min.js`

### 3.2. Repo có một nhóm file legacy không parse được bằng Prettier

Đây không phải bug mới do formatter tạo ra. Đây là code/HTML cũ vốn đã không hợp lệ theo parser hiện đại, nhưng trước đó chưa bị soi tới.

Những file này phải được exclude có chủ đích để đạt mục tiêu thực tế hơn:

> `npm run format` phải hoạt động ổn định trên phần còn lại của codebase, thay vì fail cả repo vì một nhóm file legacy đã biết.

Kết quả cuối cùng là:

- Prettier có thể chạy trên toàn repo theo nghĩa thực tế
- các file artifact/minified không bị rewrite vô ích
- các file legacy parse lỗi không còn chặn pipeline format

Và quan trọng hơn: format bây giờ là một command có thể dùng thật, không phải một nút trang trí trong `package.json`.

## 4. Test xanh lại sau một ngày dọn và sửa

Đây là phần quan trọng nhất.

Một ngày như hôm nay rất dễ rơi vào bẫy “dọn xong nhưng không chắc còn đúng”. Vì vậy điều có ý nghĩa không phải là đã sửa bao nhiêu file, mà là cuối ngày repo có còn giữ được hành vi không.

Kết quả cuối cùng: `npm test` xanh hết.

Điều đó có nghĩa là ba việc rất khác nhau đã cùng đứng vững sau thay đổi:

- test stabilization và cleanup trước đó không bị phá lại
- source tree mới sau khi flatten vẫn build đúng
- repo-wide formatting setup không làm vỡ luồng test/build

Nói ngắn gọn: đây không phải một ngày “trông có vẻ sạch hơn”. Đây là một ngày mà hệ thống chịu được thay đổi mạnh tay nhưng vẫn giữ được verification cuối cùng.
