## 1. Scan vô hạn — bandwidth ngốn không dừng

### Vấn đề

`scheduleScan()` chạy mỗi 10 phút mãi mãi dù không tìm thấy peer nào. Mỗi cycle gửi ~200 HTTP probe. Với 10 peer cùng làm điều này → hàng nghìn request vô nghĩa/giờ.

### Fix

**`scan()` — dừng sớm khi đủ peers** (`maxFound = MFND = 10`):

```js
// lib/scan.js
const MFND = 10   // mfnd = max found per cycle
// Worker task sets stp = true once found.length >= mfnd
// Remaining queued tasks check: if (stp) return null
```

**`schd()` — exponential backoff khi cycle trống**:

```
Empty cycle 1  →  next scan sau 20 phút  (siv = siv * 2)
Empty cycle 2  →  sau 40 phút
...
Cap            →  2 giờ (MSIV)
Discovery      →  reset về 10 phút ngay lập tức
Peer disconnect (bye) →  reset và scan lại ngay
```

Backoff theo dõi bằng cờ `fic` (foundInCycle). Đặt `fic = false` đầu mỗi `schd()` tick; `adp()` đặt `fic = true` khi thêm peer thành công. Sau 2 phút kiểm tra: nếu `!fic` → tăng `siv`.

## 2. Full-mesh — tất cả kết nối tất cả, mạng tự chết

### Vấn đề

AXE MOB chỉ giới hạn **inbound** — khi `mesh.near >= opt.mob`, AXE redirect peer mới đến nơi khác. Nhưng **outbound** từ scan (`pmsh.hi()`) không có giới hạn. Nếu scan tìm được 100 peers và cố kết nối tất cả → O(n²) links → mạng chết.

### Fix

**`adp()` kiểm tra `r.axe.up` trước khi gọi `pmsh.hi()`**:

```js
const MUPS = 10
// adp = add peer
function adp(url) {
  ...
  const ups = Object.keys(r.axe.up || {}).length
  if (pmsh && ups < MUPS) pmsh.hi({ id: url, url, retry: 9 })
  // PEX broadcast vẫn xảy ra dù ups >= MUPS
}
```

`r.axe.up` là object trong `lib/axe.js` tracking các kết nối upstream đang hoạt động. Không cần thêm counter riêng — nguồn truth duy nhất.

**Hai lớp giới hạn topology**:

| Layer | Cơ chế | File |
|-------|--------|------|
| Inbound | AXE MOB — redirect khi `mesh.near >= opt.mob` | `lib/axe.js` ~491 |
| Outbound | `MUPS = 10` — scan không gọi `hi()` nếu đã đủ | `script/server.js` `adp()` |

## 4. Peer Exchange (PEX)

Khi peer mới kết nối:

```js
// server.js — "hi" event
pmsh.on("hi", peer => {
  // Gửi danh sách peers đang biết
  pmsh.say({ dam: "pex", peers: knownUrls }, peer)
})

// Handler nhận pex
pmsh.hear["pex"] = (msg, _peer) => {
  msg.peers.forEach(url => adp(url))
}
```

Khi tìm thấy peer mới qua scan, broadcast ngay cho tất cả peers đang connected — không đợi đến lần kết nối tiếp theo.

PEX **không dùng public graph** — giao tiếp trực tiếp qua DAM `"pex"` message. An toàn với graph có policy bảo mật.

## 5. Self-discovery (`lib/discover.js`)

Peer cần biết domain/IP của mình để build URL tự giới thiệu. Ưu tiên offline-first:

1. `~/.config/zen/domain` — file config do install.sh hỏi khi setup
2. `ip route get 8.8.8.8` → parse output lấy `src <IP>`
3. `hostname -I` → lấy IP đầu tiên không phải loopback
4. STUN `stun.l.google.com:19302` — RFC 5389 Binding Request → XOR-MAPPED-ADDRESS
5. `api.ipify.org` / `ifconfig.me` — HTTP fallback

Domain được lưu lại vào `DOMF` lần đầu tìm được. Nếu vẫn chưa biết khi server start, `ldom()` trong `server.js` bắt domain từ `Host` header của request đến đầu tiên.

## 6. Cấu trúc scan thông minh

Từ domain `peer1.akao.io`, scanner tự suy ra pattern:

```
mkpat("peer1.akao.io") → { prefix: "peer", index: 1, suffix: ".akao.io", padLen: 0 }
bdom(pat, 0)  → "peer0.akao.io"
bdom(pat, 2)  → "peer2.akao.io"
bdom(pat, 15) → "peer15.akao.io"
```

Với zero-padding (`relay01.host`):
```
mkpat("relay01.host") → { prefix: "relay", index: 1, suffix: ".host", padLen: 2 }
bdom(pat, 5) → "relay05.host"
```

Probe song song (`CONC = 10`), HTTPS trước HTTP, timeout 3s. Dừng sau `MFND = 10` peers.

## 10. Browser AXE — cải thiện peer discovery trên browser

### Vấn đề cũ

Browser trước đây chỉ kết nối thụ động — chờ relay push data. Không biết có relay nào khác ngoài URL được truyền vào `peers:[]`. Không tự tìm thêm relay.

### Các cải thiện

**`/peers` JSON endpoint** — relay expose danh sách peers hiện biết:

```http
GET https://peer0.akao.io:8420/peers
→ ["wss://peer0.akao.io:8420/zen","wss://peer1.akao.io:8420/zen"]
```

Response CORS-enabled, trả JSON array các URL. Browser fetch endpoint này từ relay đang kết nối, rồi kết nối thêm vào các relay khác trong danh sách.

**Browser `bscan()`** — khi biết domain relay (`peer1.akao.io`), browser tự suy pattern và scan sibling relay tương tự Node.js:

```
peer1.akao.io → thử peer0.akao.io, peer2.akao.io, peer3.akao.io...
```

Dùng `lib/scan.js` qua fetch thay vì TCP probe. Tìm được relay mới → kết nối ngay.

**BroadcastChannel discovery** — các tab trong cùng origin share peers với nhau:

```js
var bc = new BroadcastChannel("zen/peers")
bc.onmessage = e => e.data.peers.forEach(url => adp(url))
bc.postMessage({ peers: knownUrls })
```

Tab 1 kết nối relay A, broadcast cho tab 2 → tab 2 kết nối relay A mà không phải scan.

## 11. Hardware peer ID — deterministic, high-entropy, restart-stable

### Vấn đề

`opt.pid` cũ là UUID ngẫu nhiên mỗi lần khởi động. Sau restart, peer có pid khác → các peer khác không nhận ra → phải rebuild trust.

### Giải pháp

`lib/discover.js` export `hwid()` — thu thập entropy cố định từ phần cứng:

| Nguồn | Ví dụ |
|-------|-------|
| `/etc/machine-id` | `6c7a8b9c0d1e2f3a4b5c6d7e8f9a0b1c` |
| MAC address (non-lo) | `02:42:ac:11:00:03` |
| Hostname | `peer1` |

Ba chuỗi này concat → `ZEN.hash(combined, null, null, {name:"SHA-256"})` → base62 → **deterministic 44-char pid**.

```js
// script/server.js
import { hwid } from "./lib/discover.js"
const hid = await hwid()
opt.pid = hid  // stable across restarts
```

**Entropy**: machine-id đã là 128-bit random (UUID). MAC thêm entropy thiết bị. Hostname thêm human label. Kết hợp → SHA-256 → không thể đoán, không trùng giữa máy, nhưng luôn giống nhau trên cùng máy.

## 12. WebRTC production-ready — browser-to-browser P2P

### Triết lý

> **Relay nên là nơi peer gặp nhau, không phải nơi peer nói chuyện mãi mãi.**

Sau khi hai browser biết nhau qua relay, mọi message nên đi trực tiếp qua DataChannel — relay bị bypass hoàn toàn. Đây mới là P2P thật sự.

### Những gì sai với `lib/webrtc.js` cũ

| Lỗi | Mô tả |
|-----|-------|
| `var zen` undefined | Dòng 186 dùng `zen` chưa khai báo → crash ngay khi load |
| Deprecated callback API | `createOffer(fn, fn, opts)` — Chrome bỏ hơn 2 năm trước |
| Signaling qua public graph | Viết offer/answer vào `/RTC/<room>` — privacy leak; không hoạt động với PEN policies |
| Chỉ có STUN | STUN chỉ work qua ~60% NAT; TURN bắt buộc cho production |
| Không có TURN | → thất bại hoàn toàn qua symmetric NAT (4G, corporate WiFi) |
| Double-offer bug | Cả hai bên tạo offer đồng thời → WebRTC crash |
| `<script>` không có `type=module` | `import` statement trong script tag thường → SyntaxError, không load được |

### Rewrite hoàn toàn

**ICE servers** — 3 STUN + 3 TURN (Open Relay Project, free):

```js
opt.rtc.ice = [
  { urls: "stun:stun.l.google.com:19302" },
  { urls: "stun:stun1.l.google.com:19302" },
  { urls: "stun:stun.cloudflare.com:3478" },
  { urls: "turn:openrelay.metered.ca:80",  username: "openrelayproject", credential: "openrelayproject" },
  { urls: "turn:openrelay.metered.ca:443", username: "openrelayproject", credential: "openrelayproject" },
  { urls: "turns:openrelay.metered.ca:443",username: "openrelayproject", credential: "openrelayproject" },
]
```

TURN cover được symmetric NAT (~85%+ NAT traversal thay vì ~60%).

**Signaling qua `dam:"rtc"` với `to: pid`** — không dùng public graph:

```js
// Signal chỉ đến đúng target
var msg = { "#": String.random(9), dam: "rtc", ok: { rtc: { offer, id: opt.pid, to: pid } } }
root.on("out", msg)
```

Relay (`lib/axe.js`) route `dam:"rtc"` trực tiếp đến peer có `p.pid === toPid` — không broadcast:

```js
function rtcway(msg) {
  var toPid = msg.ok && msg.ok.rtc && msg.ok.rtc.to
  if (!toPid) return false
  for (var id in opt.peers) {
    var p = opt.peers[id]
    if (p && p.pid === toPid && p.wire) { mesh.say(msg, p); return true }
  }
  return false
}
```

**DataChannel → mesh peer** — sau khi DataChannel mở, relay bị bypass:

```js
dc.onopen = function () {
  pc.id   = pid
  pc.say  = function (raw) { dc.send(raw) }  // mesh dùng pc.say để send
  pc.wire = dc
  mesh.hi(pc)  // join mesh — relay không còn on đường nữa
}
```

**Deterministic initiator** — lower pid luôn initiate, loại bỏ double-offer:

```js
opt.rtc.connect = function (pid) {
  if (opt.pid >= pid) return  // other side will initiate
  mkpc(pid, false)             // we create offer
}
```

**ICE restart on failure**:

```js
pc.oniceconnectionstatechange = function () {
  if (pc.iceConnectionState === "failed" && pc.restartIce) pc.restartIce()
}
```

**`bpids` trong PEX** — relay gửi danh sách browser peers (peers có pid nhưng không có url):

```js
// script/server.js — "hi" handler
var bpids = Object.values(opt.peers)
  .filter(p => p.pid && !p.url)
  .map(p => p.pid)
mesh.say({ dam: "pex", peers: [...kprs], bpids }, peer)
```

Browser nhận `bpids` → tự gọi `opt.rtc.connect(pid)` cho từng browser peer:

```js
// lib/webrtc.js — wraps mesh.hear["pex"]
mesh.hear["pex"] = function (msg, from) {
  if (_pex) _pex(msg, from)
  if (Array.isArray(msg.bpids)) {
    msg.bpids.forEach(function (pid) { opt.rtc.connect(pid) })
  }
}
```

### Luồng kết nối đầy đủ

```
Browser A ──WS──► Relay ──WS──► Browser B
                         ↑
         ← pex {bpids:[B.pid]} ─┘
A: rtc.connect(B.pid) [A.pid < B.pid, A initiates]
A ── dam:rtc {offer, to:B.pid} ──► Relay ──► B
B ── dam:rtc {answer, to:A.pid} ──► Relay ──► A
A,B exchange ICE candidates via Relay
         ──── DataChannel OPEN ────
A ◄════════════ data (no relay) ════════════► B
```

## 13. Examples — fix ES module loading

**Vấn đề**: tất cả examples load `webrtc.js` bằng `<script src>` thông thường. File này dùng `import` statement → `SyntaxError: Unexpected token 'import'` trong non-module context → toàn bộ script không chạy.

**Fix**:

```html
<!-- Trước (broken) -->
<script src="/zen.js"></script>
<script src="../lib/webrtc.js"></script>

<!-- Sau (working) -->
<script type="module" src="/zen.js"></script>
<script type="module" src="../lib/axe.js"></script>
<script type="module" src="../lib/webrtc.js"></script>
<script type="module">
  import ZEN from "/zen.js";
  var zen = new ZEN({ peers: [...] });
</script>
```

Files đã fix: `examples/axe.html`, `examples/basic/meet.html`, `examples/basic/stream.html`, `examples/game/space.html`, `test/panic/axe/index.html`.

## 16. Triết lý hôm nay

### P2P thật sự, không phải hub-and-spoke

Phần lớn "P2P" thực tế vẫn là star topology — mọi browser kết nối qua một server trung tâm. ZEN muốn làm khác: relay chỉ là điểm gặp gỡ ban đầu. Sau khi hai browser biết nhau, họ nói chuyện thẳng — relay không còn trên đường dữ liệu.

### Entropy từ phần cứng — không phải ngẫu nhiên

Peer ID ngẫu nhiên tốt cho an ninh nhưng xấu cho ổn định. Hardware entropy (machine-id + MAC + hostname → SHA-256) cho ID ổn định qua restart mà không cần lưu file hay nhớ password. Deterministic = reproducible = debuggable.

### 3–6 ký tự — toán học không cần văn xuôi

Code phải đọc như toán, không như văn xuôi. `adp` < `addPeer`. `fic` < `foundInCycle`. Khi đủ context (cùng file, cùng domain), 3 ký tự là đủ. Giảm noise, tăng tín hiệu.

### Docs song song với code

Không để docs lag sau code. Mỗi feature commit kèm diary + doc update trong cùng ngày. Không để khoảng cách giữa implementation và documentation.
