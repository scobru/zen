## 2. Refactor MCP: boot/attach pattern

Trước đây `lib/mcp/server.js` gộp mọi thứ vào một hàm `start()` cồng kềnh. Đã tách ra thành 3 hàm rõ ràng:

- **`boot()`** — tạo ZEN instance, load hardware identity, trả về `{ zen, hwIdentity, initialPeers }`
- **`attach(zen, opts)`** — mount toàn bộ MCP logic lên ZEN instance đã có sẵn
- **`start()`** — boot() + attach() cho trường hợp standalone

Lý do: muốn relay (`script/server.js`) có thể gọi `attach(zen, { hwIdentity })` và biến relay trở thành MCP server luôn — không cần process thứ hai.

## 3. Nhúng MCP vào relay qua IPC socket

**Thay đổi kiến trúc quan trọng nhất trong session này.**

Trước: MCP server là một process riêng, kết nối vào relay qua WebSocket peer-to-peer. Phức tạp, có latency, cần quản lý hai process.

Sau: MCP server nhúng trực tiếp vào relay worker qua Unix IPC socket (`/tmp/zen-mcp-{port}.sock`).

Cách hoạt động:
- Khi relay khởi động, nếu chạy trong môi trường Node.js, nó tự gọi `import("./service.js")` (lazy load)
- `lib/service.js` gọi `attach(root, { hwIdentity, ipc: true })`
- MCP server lắng nghe trên Unix domain socket
- `lib/mcp.js` (client bridge) detect IPC socket, kết nối qua đó thay vì stdio
- IDE/agent gọi `npx @akaoio/zen mcp` → bridge tới relay qua IPC

**Ưu điểm:**
- Một process duy nhất: relay IS the MCP server
- Không có WebSocket peer phụ làm rối topology
- Browser peer không load MCP (check `Zen.window`)
- Thiết kế giống Gun/SEA: import zen trên, import mcp dưới

Các files thay đổi chính: `lib/mcp/server.js`, `lib/mcp.js`, `script/server.js`, `lib/service.js`.

## 7. Cuộc chiến với suitePush (zen.push qua relay chain)

### Vấn đề

`zen.push(pubB, data)` gửi ephemeral DAM relay message từ browserA (kết nối zen) tới browserB (kết nối peer1). Phải đi qua chain zen → peer0 → peer1.

Test pass khi chạy riêng lẻ (`--suite=push`), nhưng fail ~65% khi chạy toàn bộ (`--suite=all`).

### Debug hành trình 1: ZEN phải có graph activity trước

Phát hiện: ZEN không gửi HI message (và do đó không có `peer.pub` được set) cho đến khi có ít nhất một GET/PUT activity. Fix: thêm handshake GET trước khi push:

```js
browserA.get("_push-handshake").once(() => {});
browserB.get("_push-handshake").once(() => {});
await sleep(3000);
```

### Debug hành trình 2: zen client cũ (từ suiteChain) làm nhiễu routing

`suiteChain` tạo `zen` client kết nối relay. Sau khi chain test xong, `zen.off()` chỉ đóng logic của ZEN nhưng không đóng WebSocket. Client cũ còn hanging peer làm relay think có thêm peer với pub cũ. Fix: tạo helper `killZen(z)` — đóng wire trực tiếp trước khi gọi `off()`.

### Debug hành trình 3: WebSocket 68s timeout

Sau suiteChain, relay WebSocket connection (68s inactivity) có thể timeout trước khi suitePush bắt đầu. Fix: thêm **30s keepalive ping** trong `src/websocket.js` (outbound connections):

```js
wire.onopen = function() {
  peer._keepalive = setInterval(() => mesh.ping(peer), 30000);
};
wire.onclose = function() {
  clearInterval(peer._keepalive);
};
```

Commit: `7ca8959`

### Debug hành trình 4: peer.wire = null khi bye

Khi một peer disconnect, `mesh.bye()` xóa peer khỏi `opt.peers` nhưng không clear `peer.wire`. Nếu relay cố route message tới peer này trong cùng event loop tick, nó vẫn thấy `peer.wire` còn và gửi → crash hoặc lost message. Fix: `peer.wire = null` ở đầu `mesh.bye()`.

Commit: `8343195`

### Debug hành trình 5: zen.js bundle vs src/ files

Phát hiện quan trọng nhất trong quá trình debug: **relay không dùng `src/mesh.js` trực tiếp, mà dùng file bundle `zen.js` đã build sẵn**.

Flow: `script/server.js` → `lib/server.js` → `import('../zen.js')` (bundle).

Hệ quả: mọi thay đổi trong `src/` không có tác dụng trên relay cho đến khi chạy `npm run build:zen` và deploy lại `zen.js`. Đã thêm `[RELAY-DBG]` và `[BYE]` logs vào `src/mesh.js` nhưng chúng không xuất hiện trong journalctl vì relay vẫn dùng bundle cũ.

Sau khi rebuild và deploy, debug logs xuất hiện.

### Debug hành trình 6: AXE PID-sort topology collapse

Từ `[BYE]` logs ngay sau relay restart, phát hiện topology collapse:

```
zen:   drops outbound to peer0 (noRec=true)
peer0: drops BOTH outbound AND inbound to peer1 (noRec=true)
peer1: drops inbound from zen (noRec=true)
```

Sau AXE PID-sort, topology ổn định:
- zen: có inbound từ peer0 và peer1 (qua DNS scan của peer1)
- peer0: **CHỈ có zen** — không có kết nối nào tới peer1
- peer1: có outbound tới zen

AXE PID-sort là cơ chế dedup connection: khi hai relay cùng kết nối tới nhau (A→B và B→A), AXE so sánh `opt.pid` (random 9-char per process) và giữ một trong hai. Vấn đề: với chain topology, peer0 luôn mất kết nối tới peer1 sau AXE.

### Debug hành trình 7: DHT routing chọn sai peer

Từ `[RELAY-DBG]` log tại zen:

```
[RELAY-DBG] to=0YscMXGD ttl=3 next=0bUGKvFX livePeers=[0bUGKvFX(w1),0kiu2ZCd(w1),0kiu2ZCd(w1),0LkvOoeu(w1)]
```

zen có kết nối trực tiếp tới peer1 (`0kiu2ZCd`), nhưng DHT vẫn chọn peer0 (`0bUGKvFX`). Lý do:

1. DHT k-buckets (trong `lib/dht.js`) chỉ add peers có `peer.url` (outbound connections). Peer1 là **inbound** tại zen → không có `peer.url` → **không được add vào k-buckets**.
2. `closest()` trả về chỉ peer0 (entry duy nhất trong k-bucket).
3. Zen gửi relay message tới peer0.
4. Peer0 route về zen (chỉ có zen trong livePeers của peer0).
5. Zen nhận lại với cùng `#` → **dedup drops** → message mất.

Vòng lặp: zen → peer0 → zen (dropped) → message không bao giờ tới peer1/browserB.

### Fix cuối cùng: flood relay messages

Thay vì route single-hop XOR-closest, chuyển sang **flood tới tất cả connected peers** (trừ sender):

```js
// BEFORE:
var next = mesh.route(msg.to, peer);
if (next) { mesh.say(fwd, next); }

// AFTER:
for (var fk in peers) {
  var fp = peers[fk];
  if (fp && fp.pub && fp.wire && fp !== peer) { mesh.say(fwd, fp); }
}
```

**Tại sao flooding an toàn:**
- Dedup bằng `msg["#"]` (preserved qua các hop) ngăn true loops
- TTL decrement giới hạn số hop
- Với 3 relays: zen gửi tới peer0+peer1 đồng thời → peer1 deliver trực tiếp tới browserB → peer0 bounce về zen nhưng zen dedup drop → xong

**Tại sao flooding đúng đắn hơn XOR routing cho ephemeral push:**
- XOR routing phù hợp với Kademlia lookup (tìm data stored at closest node)
- Ephemeral push cần gửi tới node đang có kết nối với recipient, không phải node XOR-closest
- Relay mesh nhỏ (vài chục node), flooding cost là O(n) per hop, acceptable

Kết quả: **14/14 tests pass**.

## Bài học rút ra

1. **Bundle vs source**: relay dùng `zen.js` bundle, không phải `src/`. Mọi debug log trong `src/` vô nghĩa cho đến khi rebuild. Từ nay sẽ check bundle date trước khi debug.

2. **AXE topology là unpredictable**: `opt.pid` random mỗi lần process restart → PID-sort kết quả thay đổi mỗi lần deploy → topology khác nhau mỗi lần. Cần thiết kế routing không phụ thuộc vào topology cố định.

3. **Dedup + single-hop routing = silent message loss**: Khi route sai peer, dedup ngăn retry → message mất không trace. Flooding giải quyết cả hai vấn đề.

4. **Inbound peers invisible to DHT**: k-buckets chỉ track outbound (có URL). Peers kết nối vào mình (inbound) không được xét trong routing. Cần nhớ khi debug routing issues.

5. **IPC > subprocess cho MCP**: embed MCP vào relay process sạch hơn nhiều. Không có WebSocket peer phụ làm nhiễu topology, không cần quản lý process lifecycle.

### 9. BOOT peer watchdog (commit `0b436ef`)

**Vấn đề:** Sau khi fix flooding và deploy, phát hiện peer0 thường xuyên mất kết nối tới peer1 sau mỗi lần restart. AXE PID-sort tombstone URL của peer1, preventing reconnect. Kết quả: mesh có thể bị isolated (peer0 không reach được peer1) cho đến lần restart tiếp theo.

**Giải pháp:** Thêm watchdog 30 giây trong `script/server.js`:

```js
setInterval(() => {
  for (const url of peers) {              // peers = BOOT peers từ env
    const norm = normalizeUrl(url);
    const knownPub = bootPubMap[norm];    // pub đã học được từ lần connect trước
    const alive = knownPub
      ? Object.values(opt.peers).some(p => p && p.wire && p.pub === knownPub)
      : opt.peers[norm]?.wire;            // fallback: outbound URL check
    if (!alive) {
      clearTombstone(norm);               // xóa tombstone + backoff counters
      pmsh.hi({ id: norm, url: norm, retry: 9 });
    }
  }
}, 30 * 1000).unref();
```

**Tại sao check by pub thay vì URL:** AXE có thể giữ inbound connection (không có URL key) trong khi tombstone outbound URL. Nếu check theo URL, watchdog nghĩ peer không connected và reconnect liên tục. Check theo pub thấy inbound connection → correct answer.

**`bootPubMap`:** Map từ normalized URL → pub, được cập nhật trong `root.on("hi")` mỗi khi peer kết nối. Watchdog dùng map này để lookup pub của BOOT peer.

**Kết quả sau deploy:** Connectivity log từ peer0 sau restart:
```
[BOOT-WATCHDOG] Reconnecting lost BOOT peer: https://peer1.akao.io:8420/zen
```
Sau một reconnect duy nhất, kết nối ổn định — không còn chu kỳ BYE liên tục.

### 10. Memory leak fixes (commit `b609c53`, issue #23)

Phân tích kỹ ba memory leaks trong relay. Tất cả đều tích lũy theo thời gian và contributes vào OOM 2GB lịch sử.

**Leak 1: `meta.raw` không có size cap (`src/mesh.js`)**

```js
// Trước (limit bị comment out):
meta.raw = raw;   // lưu toàn bộ raw string vào dup map, không giới hạn size

// Sau (re-enable 96KB cap):
if (raw.length < 96e3) { meta.raw = raw; }
```

Raw strings lớn hơn 96KB không được cache trong dup map. Các message lớn (PUT của binary data, sync payload) không bị giữ 9 giây trong dup map.

**Leak 2: `dup.js` LRU eviction không được enforce**

```js
// Trước: opt.max = 999 defined nhưng không dùng
function s(id, ttl) {
  var k = s.s.get(id);
  if (k) { ... return k; }
  s.s.set(id, (k = { v: 0 }));
  // Không check size!
  ...
}

// Sau: evict oldest khi đầy
if (s.s.size >= opt.max) {
  s.s.delete(s.s.keys().next().value);  // oldest = first in insertion-ordered Map
}
```

Với `opt.max = 999`, Map có thể không bao giờ đầy trong thực tế (messages expire sau 9s), nhưng dưới tải cao hoặc spam messages, nó có thể grow vô hạn.

**Leak 3: `peer.queue` accumulate sau disconnect (`src/mesh.js`)**

```js
// Trước: mesh.bye không clear queue
function bye(peer, ...) {
  peer.wire = null;
  // peer.batch, peer.tail, peer.queue còn lại
}

// Sau: clear buffers explicitly
mesh.bye = function(peer, ...) {
  peer.batch = peer.tail = peer.queue = null;
  peer.wire = null;
  ...
};
```

Khi peer disconnect, `peer.wire = null`. Mỗi lần `send(msg, peer)` được gọi sau đó, nó vào catch block → `peer.queue.push(raw)` → queue tích lũy raw strings. Peer object bị giữ trong `opt.peers` một thời gian → leak.

**Leak 4 (documented, chưa fix): `root.graph` infinite growth**

`root.graph = {}` là object accumulate MỌI dữ liệu PUT đã nhận. Không có TTL, không có eviction. Đây là root cause thực sự của OOM 2GB. Cần design decision trước khi fix — documented tại issue #23. Options:
- Relay-only mode: relay không lưu graph data (forward only)
- LRU với max size budget
- External storage adapter thay vì in-memory object

**GitHub issue #23** created với full analysis của cả 4 leaks.

### 11. UDP unicast fast path cho inter-relay relay messages (commit `ba79846`)

Đây là tính năng quan trọng nhất trong ngày.

**Motivation:** `zen.push` và các DAM relay messages đang đi qua WebSocket (TCP + TLS + WS framing). Giữa các relay server (public IP, không có NAT), UDP thuần là tốt hơn: latency thấp hơn, overhead ít hơn, không có head-of-line blocking.

**Design constraint:** Không cần QUIC, không cần NAT traversal, không cần DTLS. Ba relay VPS có public IPv4 → raw `dgram` UDP đủ rồi. Security sẽ bổ sung sau (Phase 2).

**Cách hoạt động:**

1. **Handshake discovery**: `dam:"?"` message được mở rộng với field `udp: <port>`:
   ```js
   // src/mesh.js - mesh.hi() send:
   var hiMsg = { dam: "?", pid: root.opt.pid, pub: opt.pub || "" };
   if (opt.udpPort) { hiMsg.udp = opt.udpPort; }
   
   // src/mesh.js - mesh.hear["?"]:
   if (msg.udp && !peer.udpPort) { peer.udpPort = msg.udp; }
   ```

2. **IP resolution hook**: `mesh.hear["?"]` được wrap trong `script/server.js`. Sau khi original handler chạy (và `peer.udpPort` được set), `setupUdpForPeer(peer)` được gọi:
   ```js
   function setupUdpForPeer(peer) {
     if (!peer || !peer.udpPort || peer.udpSay) return;
     let udpAddr = null;
     // Inbound: lấy IP từ underlying socket
     if (peer.wire?._socket) {
       udpAddr = peer.wire._socket.remoteAddress;
       if (udpAddr?.startsWith('::ffff:')) udpAddr = udpAddr.slice(7);
     }
     // Outbound: extract hostname từ peer.url
     if (!udpAddr && peer.url) {
       const m = peer.url.match(/(?:wss?|https?):\/\/([^:/[\]]+)/);
       if (m) udpAddr = m[1];
     }
     if (udpAddr) {
       udpPeerMap[udpAddr] = peer;
       peer.udpAddr = udpAddr;
       peer.udpSay = (fwd) => {
         const buf = Buffer.from(JSON.stringify(fwd), 'utf8');
         udpSock.send(buf, peer.udpPort, udpAddr, () => {});
       };
     }
   }
   ```
   
   **Lý do wrap ở server.js chứ không phải src/mesh.js:** `peer.udpPort` chỉ available AFTER `mesh.hear["?"]` handler chạy. `root.on("hi")` fire TRƯỚC khi `dam:"?"` round-trip hoàn thành (được gọi trong `mesh.hi` ngay khi tạo peer, không phải sau handshake). Wrapping `mesh.hear["?"]` là correct timing.

3. **Fast path trong relay handler** (`src/mesh.js`):
   ```js
   // mesh.hear["relay"] - trước khi flood:
   var fwdRaw;
   for (var fk in peers) {
     var fp = peers[fk];
     if (fp && fp.pub && fp.wire && fp !== peer) {
       if (fp.udpSay) {
         if (!fwdRaw) { try { fwdRaw = JSON.stringify(fwd); } catch(e) {} }
         if (fwdRaw) { try { fp.udpSay(fwdRaw); continue; } catch(e) {} }
       }
       mesh.say(fwd, fp);  // fallback to WebSocket
     }
   }
   ```
   
   `fwd` là object với decremented TTL — phải serialize lại, không thể dùng `msg._.raw` (TTL không giảm). Serialize một lần (`fwdRaw`), reuse cho tất cả UDP peers.

4. **Incoming UDP**: `udpSock.on('message')` lookup peer trong `udpPeerMap` bằng `rinfo.address`, gọi `pmsh.hear(raw, peer)` — giống hệt như WS message.

5. **Cleanup**: `root.on("bye")` xóa `udpPeerMap[peer.udpAddr]` và null `peer.udpSay`.

## 10. Fix AXE topology churn — BOOT-WATCHDOG tái kết nối vô nghĩa (tối cùng ngày)

Sau khi deploy UDP, vẫn thấy log `[BOOT-WATCHDOG] Reconnecting lost BOOT peer` trên zen mỗi 30 giây, kèm AXE conflict drop ngay sau đó. Topology thực ra ổn (peer0 có inbound tới zen), nhưng watchdog không nhận ra.

### Root cause

AXE PID-sort khi xử lý dual-connection:
- zen (PID cao nhất: `0wTw…`) drop outbound tới peer0, giữ inbound từ peer0
- Sau conflict: `axe.up[peer0.pid] = inbound_peer` (peer object không có URL)

BOOT-WATCHDOG kiểm tra:
1. `bootPubMap[norm]` → pub của peer0 — **nhưng bootPubMap chưa bao giờ được set đúng**: `on("hi")` fire khi wire mở, trước khi "?" exchange hoàn tất, nên `peer.pub = ""` tại thời điểm đó.
2. Fallback: `opt.peers[peer0.url].wire` — wire của outbound đã bị AXE close → `null`.
3. Kết quả: watchdog nghĩ peer0 không kết nối → reconnect → AXE drop lại → lặp mãi mỗi 30s.

### Fix (commit `8f355ad`)

**`script/server.js`:**

1. Trong wrapper `mesh.hear["?"]` (sau `_origHearQ`): cập nhật `bootPubMap` khi pub đã được set — điểm duy nhất đảm bảo `peer.pub` hợp lệ cho outbound BOOT peers.

2. Trong BOOT-WATCHDOG: ngoài `opt.peers`, còn scan `axe.up` theo pub — `axe.up[pid]` sau conflict chứa inbound peer (không có URL nhưng có pub + wire live).

```javascript
// Check axe.up (includes inbounds kept by AXE after conflict)
const axeUp = (at && at.axe && at.axe.up) || {};
const alive =
  Object.values(opt.peers || {}).some(p => p && p.wire && p.pub === knownPub) ||
  Object.values(axeUp).some(p => p && p.wire && p.pub === knownPub);
if (alive) continue;
```

### Kết quả

Sau fix: watchdog vẫn tick mỗi 30s nhưng **không log gì, không reconnect**. Quan sát 135 giây (4+ chu kỳ watchdog) trên cả 3 máy — zero churn. AXE conflict chỉ chạy **một lần** khi startup để resolve dual-connection, sau đó topology ổn định hoàn toàn.

## 11. UDP Phase 2: Security, hardening, và fix inbound fast path

### Vấn đề

UDP Phase 1 (unicast fast path giữa các relay) đã hoạt động về mặt cơ bản, nhưng còn nhiều lỗ hổng:

1. **Không có authentication**: bất kỳ ai cũng có thể inject UDP packet vào relay.
2. **Không có size guard**: packet lớn → crash hoặc memory issue.
3. **Self-connection guard thiếu**: `setupUdpForPeer` không check `peer.pid === root.opt.pid`.
4. **`adp()` kết nối tới chính mình**: DNS scan tìm ra hostname của relay → `adp()` thử kết nối → AXE drop → cascade rescan mỗi ~5 phút.
5. **`adp()` retry tombstoned peers**: các peer bị AXE tombstone vẫn được retry mỗi chu kỳ scan.
6. **Inbound không có UDP fast path sau AXE conflict**: khi AXE drop outbound, inbound surviving không được gán `udpSay` → relay message vẫn qua WS.

### Giải pháp

#### Token authentication (commit `809e1ec`)

Mỗi lần relay khởi động, tạo một `UDP_TOKEN` ngẫu nhiên (32 hex chars từ `crypto.randomBytes(16)`). Token này được trao đổi qua TLS WebSocket trong handshake `dam:"?"`. Mỗi UDP packet phải bắt đầu bằng `remoteToken + '|'`; bất kỳ packet nào không match `UDP_TOKEN + '|'` đều bị drop.

```javascript
const UDP_TOKEN = crypto.randomBytes(16).toString('hex');
// send: remoteToken + '|' + JSON.stringify(fwd)
// recv: if (!raw.startsWith(UDP_TOKEN + '|')) return;
```

#### Size guard, self-connection guard

```javascript
if (buf.length > 60000) return; // 60KB limit
if (peer.pid === root.opt.pid) return; // skip self
```

#### `adp()` guards (commit `f399eb8`)

```javascript
// Skip own URLs (DNS scan tìm ra hostname của chính mình)
if (url === surl || url === surl6 || ...) return;
// Skip tombstoned peers (AXE đã drop vĩnh viễn, BOOT-WATCHDOG handle riêng)
if (tombs.has(url) || tombs.has(normUrl) || ...) return;
```

#### Fix inbound UDP fast path (commit `0ef59f4`)

**Root cause**: `setupUdpForPeer` được định nghĩa bên trong `setImmediate(() => { ... })`, nhưng `root.on("bye")` handler nằm ngoài closure đó. Khi bye fires:
1. `udpPeerMap[peer.udpAddr]` bị xóa, `peer.udpSay = null`.
2. Inbound surviving nằm trong `root.axe.up` (keyed by PID), không có `udpSay`.
3. Fix: sau khi xóa UDP entry, iterate `root.axe.up` và gọi `setupUdpForPeer()` cho tất cả surviving peers.
4. `setupUdpForPeer` được chuyển ra outer scope (ngoài `setImmediate`) để bye handler có thể gọi được.

```

22:32:43 [BYE] url=https://peer0.akao.io:8420/zen pub=0bUGKvFX noRec=true
22:32:43 [UDP] Fast path for 0bUGKvFX → peer0.akao.io:8421   ← inbound kế thừa ✅
```

### Firewall

`ufw allow 8421/udp` đã được mở trên cả 3 máy: zen, peer0, peer1.

### Kết quả (Phase 2 tính đến commit `0ef59f4`)

## 12. UDP Phase 2 hoàn chỉnh: DAM ping/pong và PEX gossip qua UDP

### Vấn đề còn lại sau commit `0ef59f4`

- **DAM ping/pong** vẫn đi qua WS: `peer.rtt` đo RTT của WebSocket, không phải UDP — routing metrics không chính xác khi UDP fast path có sẵn.
- **PEX gossip** vẫn đi qua WS: peer-list send (lên đến 50 peers) tốn WS bandwidth không cần thiết khi relay-to-relay đã có UDP fast path.

### Giải pháp (commit `21fddb4`)

#### DAM ping qua UDP

Override `mesh.ping` trong `script/server.js`: nếu `peer.udpSay` đã được set, gửi ping trực tiếp qua UDP. AXE gọi `mesh.ping(peer)` ngay khi connect và mỗi 30s — từ lần ping thứ 2 trở đi (sau khi `setupUdpForPeer` đã chạy), ping sẽ đi qua UDP.

```javascript
const _origMeshPing = mesh.ping;
mesh.ping = function(peer) {
  if (peer && peer.udpSay) {
    peer.udpSay({ dam: 'ping', t: +new Date(), '#': Math.random().toString(36).slice(2) });
    return;
  }
  _origMeshPing.call(this, peer);
};
```

#### DAM pong qua UDP

Override `mesh.hear['ping']`: nếu peer có `udpSay`, reply pong qua UDP. Pong được route qua `pmsh.hear()` trên bên nhận (cùng path như WS) → `mesh.hear['pong']` cập nhật `peer.rtt`. Kết quả: `peer.rtt` phản ánh **UDP RTT** thực sự.

#### PEX gossip qua UDP — `saySmart(msg, peer)`

Helper `saySmart(msg, peer)` ưu tiên `peer.udpSay` nếu có, fallback về `mesh.say`. Dùng cho:
- Peer-list send trong `root.on('hi')` (payload lớn nhất, lên đến 50 peers)
- PID announcement tới các peers đang kết nối

Browser peers và peers chưa có UDP fast path vẫn nhận qua WS — backward compatible hoàn toàn.
