## 2. Phân tích kiến trúc peer management — phát hiện 5 điểm drift

User nhận ra `opt.peers`, `kprs`, `axe.up`, `bootPubMap`, `bootPidMap` là 4–5 store riêng biệt đang track cùng dữ liệu theo những cách khác nhau. Phân tích cụ thể:

| Vấn đề | Mô tả |
|--------|-------|
| URL normalization | `wss↔https` inline 16+ lần trong server.js |
| 4 peer stores | `kprs`, `opt.peers`, `axe.up`, `bootPubMap/bootPidMap` |
| TTL không nhất quán | `kprs` evict sau 30 phút, `peers.json` load 7 ngày |
| `adp()` vs `adpbr()` | Hai implementations song song cùng concept |
| `kprs` bị overload | Vừa là cache vừa là source of truth |

## 4. PeerRegistry class

**Commit:** `3c123bc` — `refactor(peers): replace kprs/bootPubMap/bootPidMap with PeerRegistry`

Tạo `lib/preg.js` (~330 dòng) làm source of truth duy nhất, thay thế toàn bộ:
`kprs` Map, `bootPubMap`, `bootPidMap`, `kprsProtect` Set, và 16+ inline URL normalizations.

**Thiết kế chính:**
- Canonical key = `https://` form (khớp với `opt.peers`); PEX wire = `wss://` qua `PeerRegistry.alt()`
- `registry.protect(peers)` — đánh dấu BOOT peers không bao giờ bị evict
- `registry.confirm(url, {pub, pid})` — lưu identity cho WATCHDOG
- `registry.bootEntries()` / `registry.confirmedNonBoot()` — dùng trong WATCHDOG
- `registry.pexList()` — cho PEX output
- `registry.load(adpFn)` / `registry.save()` — persistence
- `registry.setSelf()` — loại khỏi PEX list và persistence

**Lý do không dùng tên `mesh`:** Dòng 721 có `const mesh = root.opt && root.opt.mesh` trong callback → shadow + self-assignment. Đổi thành `route`.

Kết quả: 759 tests pass ✅, deploy 3 máy ✅.
