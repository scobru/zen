## 7. Deploy thật trên peer0 / peer1 — 6 vòng bug fixing liên tiếp

Ngay sau khi rewrite xong, deploy thực tế lộ ra hàng loạt bug mà `dash -n` không thể thấy được. Hai máy peer dùng nvm, không có Node.js system-wide, nên đây là bài test runtime rất tốt.

### Round 1 — `sudo apt-get` bị block

Chạy:

```sh
sh install.sh --yes
```

bị block vì `sudo apt-get` cần password nhưng không có terminal tương tác.

Cách xử lý: dùng `--skip-deps` vì Node đã cài qua nvm.

### Round 2 — `npm` không thấy, `systemctl` bị chặn

Chạy:

```sh
sh install.sh --skip-deps --yes
```

Kết quả:

- peer0: `npm not found`
- peer1: `sudo systemctl` blocked vì chưa có sudoers phù hợp

Root cause quan trọng:

- `nvm.sh` là script bash-specific
- source từ `#!/bin/sh` (dash) fail silently
- PATH không được cập nhật → `npm` không thấy

Fix:

1. Trong `install_zen()`, gọi npm qua bash thay vì source trực tiếp từ sh:
   ```sh
   bash -c ". ~/.nvm/nvm.sh && npm ..."
   ```
   Commit: `7a36357`
2. Bootstrap sudoers thủ công trên peer0 + peer1

### Round 3 — install bị block ngay tại `configure_sudoers()`

Sau khi đã có sudoers cho start/stop/restart, install vẫn bị block vì rule cũ chưa đủ quyền cho chính quá trình install.

Cách chạy đúng chuyển thành:

```sh
echo password | sudo -SE sh install.sh ...
```

Trong đó:

- `-S`: đọc password từ stdin
- `-E`: giữ environment, đặc biệt là `$HOME` để nvm còn hoạt động đúng

### Round 4 — peer1 OK, peer0 chết ở `Creating systemd service`

peer1 cài thành công, nhưng peer0 exit 127 ở bước tạo systemd service.

Root cause rất đáng nhớ của POSIX sh:

```sh
node_bin="$(command -v node)"
```

Khi `set -e` bật:

- trong bash: command fail bên trong assignment thường không làm script chết ngay
- trong POSIX sh / dash: script exit luôn

Fix:

```sh
node_bin=""
node_bin=$(command -v node 2>/dev/null) || true
```

sau đó mới fallback sang nvm.

Commit:

- `f4d114b` — lookup `node_bin` an toàn với `set -e`

### Round 5 — cùng bug lặp lại ở `install_autoupdate()` + lộ thêm bug `whoami`

peer0 lại chết lần nữa vì cùng pattern `node_bin` nhưng ở chỗ khác (`install_autoupdate()`). Đồng thời lộ ra bug khác:

- `$(whoami)` khi chạy dưới `sudo -SE` trả về `root`
- systemd unit bị ghi `User=root` thay vì user thật `x`

Fixes:

1. Thêm:
   ```sh
   REAL_USER="${SUDO_USER:-$(id -un)}"
   ```
   rồi thay mọi `$(whoami)` bằng `$REAL_USER`
2. Fallback tìm node bằng `find` bị lỗi vì match cả thư mục `include/node`
3. `sort -V | tail -1` còn chọn bản mới nhất cài trong nvm, không phải default đang dùng
4. Giải pháp cuối cùng: hỏi thẳng nvm default qua bash
   ```sh
   bash -c '. ~/.nvm/nvm.sh && command -v node'
   ```

Commits:

- `bc9e73a` — fix `REAL_USER`, `node_bin` path, set-e-safe `command -v`
- `228ffda` — resolve `node` theo nvm default thay vì bản mới nhất được cài

### Round 6 — install OK nhưng service fail với `EADDRINUSE`

peer0 cài xong nhưng service không start được vì port `8420` vẫn bị giữ bởi process `relay` cũ. Unit file đã bị xóa nhưng process orphan vẫn còn chạy.

Fix production:

```sh
fuser -k 8420/tcp
systemctl restart zen
```

Kết quả cuối của session 053:

- zen.akao.io: `zen.service` active ✅
- peer0.akao.io: `zen.service` active ✅
- peer1.akao.io: `zen.service` active ✅

## 8. Lessons learned lớn nhất từ deploy thật

Đây là những bài học đáng giá nhất rút ra trong hai ngày qua:

1. **`var=$(failing_cmd)` dưới `set -e` không portable**  
   Bash và dash không xử lý giống nhau. Trong shell portable, luôn viết defensive:
   ```sh
   var=""
   var=$(cmd) || true
   ```

2. **`nvm.sh` yêu cầu bash thật sự**  
   Không thể trông chờ `sh` source được nvm. Nếu script chính là POSIX sh, phải bọc phần nvm bằng `bash -c`.

3. **`sudo -SE` rất quan trọng cho deploy không tương tác**  
   `-S` để đưa password qua stdin; `-E` để giữ `$HOME`, nếu không thì nvm path sẽ sai.

4. **`$(whoami)` dưới sudo không phản ánh user deploy thật**  
   Phải dùng `${SUDO_USER:-$(id -un)}`.

5. **Xóa unit file không đồng nghĩa process đã chết**  
   Orphan process vẫn có thể giữ port và làm install mới fail bằng `EADDRINUSE`.

6. **`dash -n` chỉ kiểm tra syntax, không kiểm tra semantic runtime**  
   Script có thể pass syntax nhưng vẫn chết ở runtime do `set -e`, environment, PATH, sudo, hoặc process state.

## 9. Hôm nay — POSIX-ify nốt `ssl.sh`

Phần khó nhất của ngày hôm nay là `ssl.sh`. Script này có tới **72 bash-isms**, và khó hơn nhiều so với 4 script trước vì chứa nhiều validation logic, regex logic, và string manipulation.

Các nhóm rewrite chính:

- `grep -oP` (Perl regex) → `sed` extraction cho IPv6 detection
- `[[ ... =~ ... ]]` → `case` patterns cho validation domain / email / path
- `${DOMAIN//:/_}` → `sed 's/:/_/g'`
- `discover_local_ip6()` bỏ Perl regex, rewrite bằng `sed`
- `is_ipv6_address()` bỏ `[[ d =~ regex && d == *:*:* ]]`, thay bằng `case` trên pattern `*:*:*`

Validation:

```sh
dash -n script/ssl.sh
```

Pass ✅

Đây là phần rewrite có giá trị lớn vì `ssl.sh` là điểm entry đầu tiên của flow cài đặt. Nếu file này không portable, toàn bộ câu chuyện "one-command SSL bootstrap" sẽ đổ vỡ trên máy minimal.

## 11. `install.sh` tự động detect SSL cert

Trước đây flow cài đặt HTTPS hơi khó chịu:

1. chạy `ssl.sh` để tạo cert
2. chạy `install.sh`
3. phải truyền lại `--https-key` và `--https-cert` một cách tường minh

Hôm nay đã thêm auto-detection ngay sau argument parsing:

```sh
_zen_cfg="${XDG_CONFIG_HOME:-$HOME/.config}/zen"
if [ -z "$HTTPS_KEY" ] && [ -f "$_zen_cfg/key.pem" ]; then
    HTTPS_KEY="$_zen_cfg/key.pem"
fi
if [ -z "$HTTPS_CERT" ] && [ -f "$_zen_cfg/cert.pem" ]; then
    HTTPS_CERT="$_zen_cfg/cert.pem"
fi
```

Kết quả là flow mới gọn hơn rất nhiều:

```sh
# Bước 1: tạo cert (chỉ làm 1 lần)
sh script/ssl.sh --domain zen.akao.io --email admin@akao.io

# Bước 2: cài đặt (tự tìm key.pem/cert.pem)
sh script/install.sh --domain zen.akao.io --yes
```

Commit:

- `bc9ffdc` — POSIX-ify `ssl.sh`, `mcp.sh` và auto-detect certs trong `install.sh`

Đây là thay đổi mang tính UX nhiều hơn là kỹ thuật thuần túy, nhưng rất đáng giá: từ góc nhìn operator, flow HTTPS gần như đã trở thành zero-config.
