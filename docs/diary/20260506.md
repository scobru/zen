Ngày hôm nay là một trong những ngày debug khó nhất từ trước đến nay. Đã mất nhiều giờ truy vết từng lớp mã nguồn để tìm ra một lỗi đúng **một dòng** — nhưng là một dòng đã khiến toàn bộ hệ thống P2P không hoạt động: dữ liệu PUT vào relay không bao giờ được chuyển sang các peer khác.

## Bối cảnh

Đã có 3 máy production: `zen.akao.io`, `peer0.akao.io`, `peer1.akao.io`. Tất cả đều chạy relay ZEN. Mọi unit test đều pass. Nhưng thực tế: PUT data vào `zen` → không bao giờ đến `peer0` hay `peer1`. Dữ liệu cũng không persist trên disk ở các peer. ZEN chưa thật sự "sống".

## Hành trình debug

### 1. Kiểm tra opt.super

Giả thuyết đầu tiên: `opt.super` không được set → relay không route. Kiểm tra `lib/server.js`:

```js
Zen.on("opt", function(root) {
  if (u === root.opt.super) root.opt.super = true;
});
```

`opt.super = true` luôn được set cho mọi ZEN instance. Giả thuyết sai.

### 2. Tombstone mechanism

Kiểm tra `mesh.bye()` và tombstone logic. Thêm log `[TOMB-URL]`, `[BYE-URL]`, `[BYE-NOTOMB]`. Kết quả: tombstones hoạt động đúng — AXE loại bỏ các kết nối trùng lặp outbound/inbound theo PID sort.

### 3. Timing race

Phát hiện log `[SAY-WIRE]` chỉ xuất hiện trong vài giây đầu sau khi relay restart — trước khi tombstones được set (AXE mất ~1-2s). Sau khi tombstones set, `peer._noReconnect = true` → SAY-WIRE không còn fire. Không phải bug thường trực.

### 4. SAY-BROADCAST không bao giờ fire

Thêm log `[SAY-BROADCAST]` vào đầu broadcast loop trong `mesh.say`. Sau khi PUT và chờ đủ thời gian: log không bao giờ xuất hiện. Relay nhận PUT, lưu vào local store, nhưng **không broadcast ra bất kỳ peer nào**.

### 5. Truy vết luồng PUT

```
wire → mesh.say(msg, peer) → msg._.via = peer
→ dup.check → put(msg) → return  [bypasses at.on("out")]
→ inside put(): pop() → ham() → fire(ctx)
→ fire(): ctx.root.on("out", msg)  [registered handler: mesh.say]
→ mesh.say(msg, null)  [broadcast mode]
```

Trong `mesh.say(msg, null)`:
1. `meta.raw` chưa set → `mesh.raw(msg, null)` encode JSON → callback lại `mesh.say(msg, null)`
2. Lần 2: `meta.raw` đã set → check line 7754:

```js
if (!peer && mesh.way) {
  return mesh.way(msg);   // ← đây!
}
```

`mesh.way` tồn tại (do AXE được load) → gọi `mesh.way(msg)`.

### 6. Root cause tìm thấy

Trong `lib/axe.js`, hàm `mesh.way`:

```js
mesh.way = function (msg) {
  if (!msg) { return; }
  if (msg.get) { return GET(msg); }
  if (msg.put) { return; }           // ← DÂY LÀ LỖI
  if (msg.dam === "rtc") { return rtcway(msg); }
  fall(msg);
};
```

**`if (msg.put) { return; }`** — mọi PUT message bị drop tại đây. Không có exception, không có warning. Chỉ là `return;`.

`mesh.way` là DHT/AXE routing layer. GET được route qua DHT. RTC messages có handler riêng. Các loại message khác dùng `fall(msg)` để broadcast. Nhưng PUT thì chỉ... biến mất.

Hàm `fall()` rất đơn giản:
```js
function fall(msg) {
  mesh.say(msg, opt.peers);
}
```

---

## Fix

```js
// Trước:
if (msg.put) { return; }

// Sau:
if (msg.put) { return fall(msg); }
```

Một ký tự `fall(msg)` thay vì `return;` — đủ để broadcast PUT đến toàn bộ connected peers qua `mesh.say(msg, opt.peers)`.

**Tại sao an toàn:**
- Dedup (`dup.check(msg["#"])`) ngăn relay xử lý cùng một PUT hai lần → không có infinite loop
- `meta.via` check trong broadcast loop (`if (peer === meta.via) { return false; }`) ngăn gửi ngược lại peer gốc
- Tombstone mechanism giữ đúng 1 connection per peer-pair

## Những gì đã học

### AXE routing model

AXE chia message theo loại:
- **GET** → DHT routing (tìm peer nào có data đó)
- **PUT** → broadcast đến all peers (trước đây bị drop!)
- **RTC** → route đến peer có `pid` cụ thể
- **other** (dam messages, etc.) → `fall()` broadcast

Ý tưởng gốc của AXE: GET routing thông minh qua DHT thay vì broadcast. PUT vẫn phải broadcast vì bất kỳ peer nào cũng có thể cần nhận. Nhưng implementation đã để lại `return;` thay vì `return fall(msg)`.

### Tại sao khó tìm

1. Log chỉ thấy "relay received PUT, stored OK" — không thấy "relay did NOT broadcast"
2. `mesh.say` phức tạp: dùng cho cả incoming lẫn outgoing, có async raw encoding, broadcast loop chia nhỏ thành batches của 9
3. `mesh.way` là hidden interceptor — nằm ở line 7754 trong `mesh.say`, giữa phần raw encoding và phần broadcast loop, chỉ active khi AXE loaded
4. Unit tests toàn pass vì chúng test local node — không test multi-relay propagation

### Các peer relay connect lại sau restart

Sau khi relay restart, peer0/peer1 tự reconnect trong vòng 2-3s (inbound connections). Không cần config thêm — `reconnect()` function với 2s delay chạy ở phía peer.

## Liên quan đến akao

Đây là bug cơ bản nhất — ZEN P2P chưa thật sự hoạt động cho đến hôm nay. Mọi tính năng multi-device, multi-user của akao đều phụ thuộc vào relay propagation. Từ hôm nay:

- PUT từ device A → relay → device B nhận được
- Dữ liệu persist trên nhiều relay node
- `peer0`, `peer1`, `zen` đều là redundant storage

Bước tiếp theo: verify persistence qua restart relay, test 2-way sync (peer PUT → zen nhận), và kiểm tra PEX (peer exchange) để relay tự discover nhau.
