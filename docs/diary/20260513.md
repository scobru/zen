### Fork testing với Hardhat/Ganache

Quyết định dùng `ganache --fork` fork mainnet tại block 19,500,000. Lý do chọn số này:
- Đủ cũ để stable (không bị reorg)

Cấu hình test: script `test/chains/setup-ganache.sh` khởi động ganache fork, sau đó `evm-fork.js` connect vào.

### Dynamic tuple bug — cái bẫy nguy hiểm nhất

Khi chạy Uniswap V3 `exactInput` (multi-hop swap với `bytes path` trong struct), transaction liên tục revert với lý do không rõ ràng. Debug mất hơn 2 giờ.

Vấn đề nằm ở ABI encoder: khi encode một `tuple`, nếu tuple có field dynamic (`bytes`, `string`, `T[]`), thì tuple đó phải được encode theo kiểu dynamic (offset + data) chứ không phải inline.

```
Sai:
head: [field1] [field2 (bytes, inline)] → WRONG

Đúng (EIP-712 + ABI spec):
head: [field1] [offset to tuple] ...
tail: [tuple data: field1 field2_offset field2_data]
```

Fix: `isDynamicType(type, components)` kiểm tra đệ quy — nếu bất kỳ component nào của tuple là dynamic, tuple đó là dynamic. Sau fix, tất cả V3 tests pass.

Đây là loại bug tinh tế nhất trong ABI encoding. ethers.js handle đúng; chúng ta cũng đã handle đúng sau khi fix.

### EIP-1559 baseFee awareness

Bug thứ hai: khi fork mainnet, `eth_gasPrice` từ ganache trả về ~2 gwei (giá fork cũ), nhưng block `baseFeePerGas` là ~13.6 gwei. Transaction dùng gasPrice=2gwei sẽ bị reject ngay vì gasPrice < baseFee.

Fix: `Provider.getGasPrice()` kiểm tra `baseFeePerGas` của latest block. Nếu `eth_gasPrice < baseFee * 1.1`, trả về `baseFee * 1.2` thay thế. Có 20% buffer đủ để pass mà không quá đắt.

### Receipt status checking

Trước khi fix: `waitForTransaction` trả về receipt dù transaction revert. Điều này che giấu bugs — test pass nhưng swap thực sự không xảy ra.

Fix: Check `receipt.status === "0x0"`, throw `Error("Transaction reverted: <hash>")` với `.receipt` attached. Sau fix, mọi revert đều bị bắt ngay lập tức → dễ debug hơn nhiều.

## Nghiên cứu ethers.js v6

### Những gì ethers.js làm tốt mà chúng ta chưa có

Sau khi hoàn thiện fork tests, tôi dành thời gian đọc kỹ source code ethers.js v6. Tóm tắt những điểm tinh túy:

**1. Human-readable ABI** — Đây là killer feature của ethers.js. Thay vì JSON ABI dài dòng, dev có thể viết:
```js
["function balanceOf(address) view returns (uint256)"]
```
Parser phải xử lý: tuples, arrays, indexed keywords, location keywords (calldata/memory/storage), anonymous events, v.v.

**2. solidityPacked / solidityPackedKeccak256** — Tương đương `abi.encodePacked` trong Solidity. Cần cho CREATE2 address computation và signature construction.

**3. getCreate2Address** — Offline computation địa chỉ CREATE2. Cần cho Uniswap V2 pair address, V3 pool address.

**4. EIP-712 TypedDataEncoder** — Permit2, DAI permit, OpenSea orders đều dùng. Không có cái này là thiếu sót lớn.

**5. Multicall3** — ethers.js v6 không có built-in. Các package ethers-multicall, wagmi đều add bên ngoài. Cơ hội để zen tốt hơn.

**6. Auto-reconnect WebSocket** — ethers.js có nhưng implementation khá phức tạp. Chúng ta implement sạch hơn với explicit Map cho subscriptions.

### Những gì ethers.js làm mà chúng ta KHÔNG cần copy

- `FetchRequest` / `Network` class hierarchy — quá phức tạp, không cần
- `Signer` abstract class — zen dùng trực tiếp `Wallet`, không cần interface này
- `TypedBigNumber` — native `BigInt` đủ rồi
- ENS resolution — ngoài scope
- Provider events (không phải WS subscriptions) — không cần

## Tier 2 — Human-readable ABI

Parser phức tạp nhất trong dự án. Viết `parseAbiItem` và `parseAbi`.

### Thách thức chính

**Tuple parsing:** `(address,uint256)` phải parse thành `{ type:"tuple", components:[...] }`. Nhưng có thể có nested: `(address,(uint256,bytes))[]`. Phải parse đệ quy, không thể dùng split(",") thông thường.

Solution: `_splitTopLevel` — split bằng `,` nhưng chỉ ở level 0 (không trong `()`). Đơn giản nhưng hiệu quả.

**Array suffix:** `uint256[]`, `address[3]`, `(address,uint256)[]` — tất cả có thể có suffix. Parse suffix riêng, apply lên base type.

**indexed keyword:** chỉ có trong events. `address indexed from` → `{ name:"from", type:"address", indexed:true }`.

**Location keywords:** `calldata`, `memory`, `storage` xuất hiện trong function params nhưng không nằm trong ABI output. Strip silently.

**stateMutability:** `view`, `pure` → readonly; `payable` → payable; còn lại → `nonpayable`.

Kết quả: 100% tương thích với JSON ABI — dùng được với Contract constructor, buildCalldata, decodeReturnData, và mọi chỗ khác.

## Tier 3 — Provider Extensions

### Multicall3

Quyết định quan trọng: build multicall như first-class method `provider.multicall(calls)` thay vì utility function riêng.

Interface: mảng `{ address, abi, method, args, allowFailure? }`. Dùng `aggregate3` của Multicall3 (`allowFailure` per call). Auto-encode calldata cho từng call, auto-decode kết quả.

Điểm tinh tế: Multicall3 deploy tại địa chỉ giống nhau trên tất cả EVM chains — `0xcA11bde05977b3631167028862bE2a173976CA11`. Không cần config gì.

### WsProvider Auto-reconnect

- `_resubscribeAll()` — sau reconnect, gửi lại `eth_subscribe` cho mọi handler đã lưu
- `_destroyed` flag — `destroy()` ngăn reconnect vĩnh viễn

## Tier 4 — EIP-712 TypedDataEncoder

EIP-712 là standard phức tạp nhất đã implement. Highlights:

### Type encoding

`_encodeType("Mail", types)` phải produce:
```
Mail(Person from,Person to,string contents)Person(string name,address wallet)
```
Quy tắc: primary type trước, các referenced types được sort alphabetically. Implement bằng DFS + sort.

### hashStruct

Recursive:
- Scalar types → ABI-encoded 32 bytes
- `string`/`bytes` → keccak256
- Arrays → keccak256 của concatenated element hashes
- Nested struct → `hashStruct(type, types, value)` (đệ quy)

### recoverTypedDataAddress

Cần recover public key từ signature → derive address. `ZEN.recover` chưa available trong public API. Solution: implement inline secp256k1 math:
- `_modPow`, `_modInv` — finite field arithmetic
- `_recoverPubPoint` — recover public key point từ (r, s, v, hash)
- `_pubPointToAddress` — keccak256 của uncompressed pubkey bytes, lấy 20 bytes cuối

Verified against ethers.js `TypedDataEncoder.hash` — kết quả khớp hoàn toàn.
