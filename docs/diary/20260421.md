# 20260421

> **Ghi nhớ**: `zen.verify(sig)` không còn đòi pub nếu signature có `v` (recovery bit). Pub được khôi phục từ toán học ECDSA trực tiếp trong quá trình verify — không cần truyền thêm. Guard curve: chỉ chạy nếu `c.recoverPub` là function.

---

Hôm nay tập trung vào một cải tiến nhỏ nhưng có ý nghĩa kiến trúc rõ ràng: `zen.verify()` học cách tự tìm public key từ chữ ký, thay vì luôn đòi caller cung cấp.

---

## 1. Bối cảnh — tại sao cần verify không có pub?

Hôm qua (20260420) đã bỏ `"*"` (pub kèm theo signed value) ra khỏi format lưu trữ. Kể từ đó, pub không còn tồn tại bên cạnh sig trong graph nữa. Ai muốn biết ai đã ký thì gọi `zen.recover(sig)` — recover pub từ chữ ký.

Nhưng nếu caller chỉ muốn xác minh chữ ký hợp lệ mà không cần biết ai đã ký, họ phải gọi thêm `recover()` rồi mới truyền kết quả vào `verify()`:

```js
const pub = await zen.recover(sig)
const data = await zen.verify(sig, pub)
```

Hai bước, mà bước một chỉ để phục vụ bước hai. Lãng phí, và không tự nhiên. Câu hỏi đặt ra: tại sao `verify()` không tự làm điều đó khi không thấy pub?

---

## 2. Phân tích toán học — verify và recover có thể dùng chung kết quả không?

Luồng verify ECDSA chuẩn:

1. Parse sig → `(r, s)`
2. Hash message → `h`
3. `z = h mod N`
4. `w = s⁻¹ mod N`
5. `u1 = z·w mod N`, `u2 = r·w mod N`
6. `Q = u1·G + u2·pub`
7. Hợp lệ nếu `Q.x mod N == r`

Luồng recover ECDSA:

1. Parse sig → `(r, s)`  
2. Hash message → `h`
3. `z = h mod N`
4. Lift `R = (r, y)` từ `r` và recovery bit `v`
5. `rinv = r⁻¹ mod N`
6. `u1 = -(z·rinv) mod N`, `u2 = s·rinv mod N`
7. `pub = u1·G + u2·R`

Hai luồng khác nhau về mặt toán học — recover trả về một điểm EC (pub), verify nhân điểm với pub và kiểm tra tọa độ x. Không thể rút gọn thành một phép tính. Nhưng kết quả của recover có thể dùng thẳng làm input của verify step 6 — đây là điểm quan trọng.

Khi `pair` không được truyền: recover pub point từ `(v, r, s, h)`, rồi dùng point đó thay cho `parsePub(pub)`. Bước tiếp theo (`u1·G + u2·pt`) chạy bình thường với point vừa recover. Nếu sig hợp lệ thì `Q.x == r` — verify pass. Nếu sig giả mạo (r, s random không phải ký từ private key hợp lệ), recover sẽ trả về một point ngẫu nhiên → `Q.x ≠ r` → fail.

Đây không phải shortcut hay optimization — verify vẫn kiểm tra đầy đủ toán học. Chỉ là pub được lấy từ sig thay vì từ caller.

## 3. Guard curve — không phải curve nào cũng hỗ trợ recover

Recovery đòi hỏi:
- Curve có dạng short Weierstrass: $y^2 = x^3 + ax + b$
- Có thể solve `liftX`: từ `x` và parity bit suy ra `y` duy nhất

Điều kiện đủ để `liftX` hoạt động là $p \equiv 3 \pmod{4}$ — cả secp256k1 và P-256 đều thỏa. Nếu sau này thêm curve không thỏa (ví dụ Ed25519 dùng twisted Edwards), `recoverPub` sẽ không được implement.

Guard trong `verify()`:

```js
if (typeof c.recoverPub !== "function") {
  throw new Error("Curve does not support public key recovery");
}
```

Kiểm tra runtime thay vì hardcode tên curve — future-proof hơn, và đặt trách nhiệm lên từng curve module: nếu muốn hỗ trợ recover-based verify, implement `recoverPub`.

## 4. Thay đổi trong `src/verify.js`

Toàn bộ thay đổi nằm trong một block rẽ nhánh giữa `parsePub(pub)` và `recoverPub(msg.v, r, s, h)`:

```js
// Trước:
const pt = c.parsePub(pub);

// Sau:
let pt;
if (pub) {
  pt = c.parsePub(pub);
} else {
  if (msg.v === undefined || msg.v === null) {
    throw new Error("Public key or recovery bit (v) required");
  }
  if (typeof c.recoverPub !== "function") {
    throw new Error("Curve does not support public key recovery");
  }
  pt = c.recoverPub(msg.v, r, s, h);
}
```

Để có `msg.v` trong signature, phải dùng `zen.sign()` của zen (không phải GUN/SEA gốc) — `sign.js` đã nhúng `v` vào sig từ trước.

---

## 5. Thứ tự tính toán — `h` và `(r, s)` phải trước `pt`

Thay đổi nhỏ nhưng cần thiết: trước đây `parsePub()` gọi ngay trước khi hash và parse sig. Sau khi thêm branch recover, `h` và `(r, s)` phải được tính trước để truyền vào `recoverPub`. Code được reorder:

```
parse sig → parse message → hash → lấy (r, s) → lấy pt (pub hoặc recover) → verify
```

Thứ tự mới hợp lý hơn về logic — tất cả input cho bước verify được chuẩn bị xong rồi mới verify.
