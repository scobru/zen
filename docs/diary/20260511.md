## 2. Review GitHub Issues (03:41)

Mở danh sách issues đang open để thảo luận xem có nên làm không.

**Issue #20 — Network ID làm domain separator cho cryptography:**  
Đây là đề xuất cũ: đưa `network = 0/1` (hoặc chain ID kiểu EVM) vào các hàm cryptography như một tham số primitive để tránh replay attack cross-network.

Thảo luận và quyết định: **không làm vào lúc này.**

Lý do:
- Trên EVM, `chainId` chỉ được đưa vào chữ ký Ethereum theo EIP-155 ở lớp transaction, không phải ở lớp ECDSA thuần túy. Các hàm hash/sign/verify ở primitive level không mang network ID.
- Hệ thống ZEN đang dùng `sea.js` (bọc WebCrypto), không phải EVM cryptography.
- Network ID nếu muốn phân tách domain thì nên làm ở lớp application (soul prefix, cert policy) chứ không baked vào primitive.
- Hệ thống chưa production, đây là premature optimization.

→ **Đóng issue #20** với ghi chú giải thích, lưu để thảo luận sau.

## 3. Review báo cáo lỗi networking của Huy (03:53)

Nhận được báo cáo chi tiết từ Huy — audit ngày 2026-05-09 phát hiện 4 vùng nguy hiểm trong `mesh.js` và `websocket.js`. Bắt đầu nghiên cứu kỹ codebase để xác nhận từng lỗi.

### Lỗi #1 — `mesh.leap` race condition (`mesh.js:274`)

**Báo cáo:** `mesh.leap` được set synchronous, nhưng callback của `mesh.raw()` (JSON.stringify) là async → khi callback fire, `mesh.leap` đã bị clear → ACK drop silently.

**Sau khi audit thực tế:**  
Lỗi được Huy mô tả chính xác về mặt lý thuyết. Tuy nhiên, primary path ACK sử dụng `dup.s.get(ack).via` — được set synchronous và persistent. `mesh.leap` chỉ là fallback path khi `via` không có. Trong thực tế production (không phải high-throughput mock), path này hiếm bị trigger. Quyết định **không remove `mesh.leap`** mà thay vào đó xóa logic set/clear nó để tránh sự phụ thuộc vào timing — thay bằng cách chỉ dùng `via`.

### Lỗi #2 — Yo-list `><` inject vô điều kiện (`mesh.js:405-418`)

**Báo cáo:** Yo-list được build từ TẤT CẢ peers đang kết nối, không phải chỉ peers mà message thực sự sẽ gửi đến → receiver không forward đến các peer trong yo-list → silent data loss.

**Sau khi audit:** Đúng. Khi `mesh.raw(msg, specificPeer)` được gọi với một peer cụ thể, yo-list vẫn được gán từ toàn bộ `opt.peers`. Receiver nhận được yo-list với 5 peer khác nhưng không forward vì nghĩ những peer đó đã nhận → data loss.

**Fix:** Thêm `&& !peer` vào điều kiện set yo-list — chỉ broadcast (peer=null) mới get yo-list:
```js
if (!msg.dam && !msg["@"] && !peer) {
```

### Lỗi #3 — `_axeGuess`/`_hiGuess` tombstone không reset (`websocket.js:55-79`)

**Báo cáo:** Counter không reset khi connection ổn định → network flap tạm thời (5 lần trong 10 phút) có thể tombstone relay vĩnh viễn trong session.

**Sau khi audit:** Đúng. `wire.onopen` không có code reset counter. Relay hợp lệ có thể bị cô lập.

**Fix:** Thêm reset cả hai counter trong `wire.onopen`:
```js
peer._axeGuess = 0;
peer._hiGuess = 0;
```

### Lỗi #4 — `dup.s.delete(peer.last)` race sau handshake (`mesh.js:524,600`)

**Báo cáo:** `peer.last` bị overwrite trước khi async callback complete → delete nhầm dup entry → message mới bị xử lý hai lần.

**Fix:** Capture `peer.last` vào local variable trước khi gọi `mesh.say()`:
```js
var lastId = peer.last;
mesh.say(hiMsg, peer);
dup.s.delete(lastId); // dùng giá trị đã capture, không bị overwrite
```

## 6. Phát hiện giới hạn 128 byte của PEN VM (05:15)

Sau khi xong networking fixes, câu hỏi mới nảy sinh: PEN VM hiện tại chỉ xử lý được string tối đa **128 byte**. Liệu có thể nâng lên vô tận không?

**Phân tích kiến trúc PEN VM:**
- `src/pen.zig` — Core VM viết bằng Zig, compile sang WASM32 freestanding (zero stdlib)
- `src/pen.wasm` — binary đã compile
- `src/pen.js` — JS compiler (build bytecode) + WASM wrapper
- Giới hạn: `MAX_STR: usize = 128` → `Value.s: [128]u8` inline trong struct

**Đánh giá 2 phương án:**
- **Option A (tăng MAX_STR):** Đơn giản nhưng vẫn có giới hạn cứng, Value struct phình to
- **Option B (WASM heap):** Dùng linear memory 4MB làm string heap với bump allocator — thực sự vô tận, Value struct nhỏ hơn

→ Chọn **Option B** vì sạch hơn và mở ra khả năng xử lý chuỗi dài thực sự.

### Thay đổi `src/pen.zig`

Thay đổi kiến trúc lớn nhất trong lịch sử PEN VM:

**Value struct (trước):**
```zig
const Value = struct {
    tag: u8,
    i: i64,          // 8 bytes
    f: f64,          // 8 bytes
    s: [128]u8,      // 128 bytes!
    slen: u16,       // 2 bytes
    // total: ~160 bytes
};
```

**Value struct (sau):**
```zig
const Value = struct {
    tag: u8,
    i: i64,          // 8 bytes
    f: f64,          // 8 bytes
    sptr: u32,       // 4 bytes — offset vào str_heap
    slen: u32,       // 4 bytes — độ dài thực
    // total: 32 bytes (5× nhỏ hơn!)
};
```

**Thêm string heap:**
```zig
const MAX_STR_HEAP: usize = 4 * 1024 * 1024; // 4MB
var str_heap: [MAX_STR_HEAP]u8 = undefined;
var str_bump: u32 = 0;

pub fn resetStrHeap() void {
    str_bump = 0;
}
```

**Cập nhật toàn bộ string operations:** `vStr`, `strStartsWith`, `strEndsWith`, `strIncludes`, `strToLower`, `strToUpper`, `strSeg`, `valEqual`, `strCmp`, `toNumber`, và các opcodes `SLICE`, `SEG`, `CONCAT`, `SEGRN`, `TONUM`.

**STR opcode:** đổi từ `[0x03, u8_len, ...bytes]` sang `[0x03, ULEB128(len), ...bytes]` — hỗ trợ string inline đến hàng MB.

### Thay đổi `src/wasm.zig`

- `buf`: 64KB → 4MB (shared input buffer cho bytecode + registers)
- `parseRegs` string case: `u16` → `u32` length (4 bytes LE)
- Gọi `pen.resetStrHeap()` ở đầu `run()` và trong `free()`

### Thay đổi `src/pen.js`

- `bc.str()`: encode length bằng ULEB128 thay vì u8
- `treeskip()` case 0x03: decode ULEB128 khi skip
- `_writeReg()` strings: length 2 bytes → 4 bytes, bỏ `Math.min(..., 0xffff)` truncation

### Bug compile đầu tiên: u64 → usize cast

```
src/pen.zig:637:19: error: expected type 'usize', found 'u64'
```

`readUleb()` trả về `u64`, nhưng `pos.*` là `usize` (u32 trên wasm32). Fix:
```zig
const len: usize = @intCast(readUleb(ctx.bc, pos));
```
