## Khởi đầu: Mining cho pen PoW

Việc đầu tiên là thêm `opt.pow` vào `ZEN.hash()` để hỗ trợ đào nonce cho pen policy. Pen có thể yêu cầu một key phải có SHA-256 bắt đầu bằng một prefix nhất định (proof-of-work), và trước đây không có API nào để tìm key hợp lệ — phải tự viết loop bên ngoài.

API mới trông như thế này:

```js
const result = await ZEN.hash(
  nonce => `order:${candle}:${nonce}`,
  null, null,
  { name: "SHA-256", encode: "hex", pow: { unit: "0", difficulty: 2 } }
);
// result: { hash: "00a3f...", nonce: "G", proof: "order:1712000000:G" }
```

`data` có thể là function nhận nonce — caller toàn quyền đặt nonce ở bất kỳ vị trí nào trong chuỗi. Nếu `data` là string thì nonce tự động nối vào cuối với dấu `:` làm separator. Khi `opt.pow` được set, `hash()` gọi đệ quy chính nó với `subOpt` (opt trừ pow) để tính hash của mỗi attempt, giữ nguyên `opt.name`, `opt.encode`, và salt.

Nonce dùng base62 thay vì integer thuần: `0, 1, ..., z, 10, 11, ...` — ngắn hơn khi search space lớn, và đẹp hơn trong key (`"order:1712:G"` thay vì `"order:1712:16"`). `intToB62()` trong `hash.js` reuse `base62.ALPHA` từ `src/base62.js` để không duplicate charset string.

Dấu `:` làm separator cũng được chọn có chủ ý: `"order:G"` rõ ranh giới giữa phần cố định và nonce, không bị nhòa như `"orderG"` trước đây.

Có một câu hỏi hợp lý xuất hiện lúc này: `hash()` đã có sẵn `opt.salt` — sao không dùng nó làm nonce? Trace code cho thấy `opt.salt` hoàn toàn không được đọc ở nhánh SHA-256 — nó chỉ tồn tại trong nhánh PBKDF2. Pen verifier dùng `{ name: "SHA-256", encode: "hex" }` nên nếu mining dùng PBKDF2 với salt là nonce, output sẽ không khớp với cách pen verify. Nonce phải nằm trong data (proof), không phải trong salt — đây là lý do kiến trúc, không phải convention. Kết quả: 6 test mới thêm vào, tất cả pass.

## Compressed pub key — đập vỡ 88 chars xuống còn 45

Từ mining nhảy sang một câu hỏi lớn hơn về format của public key. Zen lúc đó lưu pub dưới dạng uncompressed: ghép x và y của điểm EC, mỗi tọa độ 44 chars base62 → tổng 88 chars. Lãng phí, bởi vì trên mọi short Weierstrass curve (secp256k1, P-256), biết x và parity của y là đủ để recover y hoàn toàn — Bitcoin và TLS đã làm vậy từ đầu.

Trước khi implement có một nhầm lẫn cần làm rõ: ban đầu tưởng zen đang dùng base64url từ GUN legacy. Đọc `src/curves/utils.js` thì thấy zen đã dùng base62 từ đầu — `decodeBase64Url` chỉ tồn tại để đọc key cũ từ GUN gốc. Đây là điểm gây nhầm lẫn cần ghi nhớ vĩnh viễn (xem note đầu trang).

Compressed format mới:

```
pub  = base62(x) + ("0" | "1")  →  45 chars  (44 chars x + 1 char parity)
epub = base62(x) + ("0" | "1")  →  45 chars
```

Parity `"0"` nếu y chẵn, `"1"` nếu y lẻ. Recovery y từ x + parity dùng công thức:

$$y = (x^3 + ax + b)^{(p+1)/4} \pmod{p}$$

Công thức này đúng cho cả secp256k1 lẫn P-256 vì cả hai đều có $p \equiv 3 \pmod{4}$. Nếu parity không khớp thì lấy $p - y$.

Tác động rõ ràng:

| Chỉ số | Trước | Sau |
|---|---|---|
| pub/epub size | 88 chars | 45 chars |
| Soul shard depth | 44 segments | 23 segments |
| Bytes per key (on wire) | 88 B | 45 B |

Thay đổi chỉ cần sửa 3 file nguồn: `src/curves/utils.js` (thêm `liftX`, sửa `pointToPub`, sửa `parsePub` với backward compat), `src/security.js` (3 chỗ hardcode `88` → `45`), `test/zen/crypto.js` (3 regex). Các file khác như `pair.js`, `sign.js`, `verify.js` gọi qua interface nên không bị ảnh hưởng. Kết quả: 400 tests pass.

## zen.recover — recoverable signature từ góc nhìn TDD

Compressed pubkey mở ra một câu hỏi tự nhiên: nếu zen đã biết cách recover y từ x, tại sao không recover cả public key từ signature? Đây là nền tảng của `ecrecover` trong Ethereum và là tính năng zen đang thiếu so với ETH/BTC. Phân tích kiến trúc cho thấy interop với chuỗi thực sự cần ba tầng:

```
Tầng 1 — Key identity:   cùng scalar secp256k1  ✅ đã có
Tầng 2 — Address format: mỗi chain encode pub key khác nhau  ✅ format.js đã có
Tầng 3 — Signing:        mỗi chain hash và serialize khác nhau  ❌ chưa có
```

Tầng 3 vẫn chưa đủ để thực sự interop với ETH hay BTC — đó là việc của session sau. Nhưng recovery bit `v` là bước đầu tiên và cũng là thứ có ích ngay lập tức: với `v`, bất kỳ ai nhận được một zen signature cũng có thể biết ai đã ký mà không cần biết pub key trước.

Implement theo TDD: viết test trước, code sau. 22 test case bao phủ:

- Cơ bản: `ZEN.recover` là function, trả về đúng pub, pub 45-char base62, sig có field `v`, deterministic
- Cross-check: backward compat với `verify`, alice và bob cho pub khác nhau
- JS types: null, true, false, 0, 1, string, object, array
- Error cases: thiếu `v`, sig tampered (trả về key sai chứ không throw — hành vi đúng về mặt toán học), callback nhận `undefined` khi lỗi
- P-256: recover hoạt động trên P-256 vì cùng $p \equiv 3 \pmod{4}$
- ZEN instance: `zen.recover` mirrors `ZEN.recover`
- Cross-curve: P-256 sig stripped `c` field → secp256k1 recovery cho pub sai; secp256k1 sig injected `c: "p256"` → P-256 recovery cho pub sai

Cần thay đổi ở 4 chỗ để test pass:

**`src/curves/utils.js`** — thêm `recoverPub(v, r, s, hashBytes)`. Toán học ECDSA recovery: $R = \text{liftX}(r, v\&1)$, sau đó $Q = u_1G + u_2R$ với $u_1 = -z \cdot r^{-1} \bmod N$ và $u_2 = s \cdot r^{-1} \bmod N$. Tất cả operations đều dùng A, B, G, N, P từ closure nên tự động đúng cho từng curve.

**`src/sign.js`** — thêm `v` vào output. Quan trọng: `v` phải tính SAU bước low-s normalization vì khi `s > HALF_N` thì `s = N - s` đồng thời flip parity `v ^= 1`. Nếu tính trước bước này sẽ ra `v` sai.

**`src/recover.js`** — module mới, đọc `msg.v` và `msg.c`, khởi tạo đúng curve instance qua `crv(msg.c || opt.curve)`, gọi `recoverPub`, encode kết quả bằng `pointToPub`.

**`src/index.js`** — expose `ZEN.recover` (static) và `zen.recover` (instance).

Một điều quan trọng phát hiện khi chạy test: cross-curve recovery không throw. `liftX` luôn tìm được điểm hợp lệ trên curve đích vì bất kỳ x nào cũng có thể là x-coordinate hợp lệ (nếu `x³+ax+b` là quadratic residue). Kết quả là key sai, không phải lỗi — bảo mật thực sự dựa vào field `c` trong sig, không phải vào throw. Test cross-curve phản ánh đúng điều này: `notStrictEqual(result, pair.pub)` thay vì `rejects`.

Sau khi rebuild: **198/198 passing**.

## base64 vs base62 cho sig — câu trả lời từ benchmark

Cuối ngày có câu hỏi: nên đổi field `s` trong signature từ base64 sang base62 không? Benchmark thực tế (50.000 lần mỗi):

```
base64: 0.17 µs/op  → 88 chars
base62: 7.00 µs/op  → 88 chars
base62 chậm hơn 41x
```

Hai encoding cho ra cùng độ dài (88 chars cho 64 bytes). base64 là C native trong Node.js; base62 dùng BigInt division — 2 lần gọi `biToB62` cho r và s riêng. Không có lý do nào để đổi: không ngắn hơn, chậm hơn 41 lần.

base62 xứng đáng cho scalar (pub, priv, epub) vì những giá trị đó làm soul path, cần ngắn nhất có thể, và chỉ encode/decode một lần khi pair được tạo hoặc load. Signature encode/decode trong mọi lần sign và verify — giữ base64 là quyết định đúng.

## Hash Mining — nonce base62, dấu `:`, và câu hỏi về opt.salt

Hôm nay thêm vào `ZEN.hash()` khả năng **đào (mine)**: loop qua các nonce cho đến khi hash đầu ra thỏa mãn yêu cầu prefix của pen PoW. Không phức tạp về mặt thuật toán — SHA-256 brute-force đơn thuần — nhưng có vài quyết định thiết kế đáng ghi lại.

## Tính năng mới: `opt.pow` trong `hash()`

API đơn giản:

```js
const result = await ZEN.hash(
  nonce => `order:${candle}:${nonce}`,
  null, null,
  { name: "SHA-256", encode: "hex", pow: { unit: "0", difficulty: 2 } }
);
// result: { hash: "00a3f...", nonce: "G", proof: "order:1712000000:G" }
```

- `data` là **function** nhận nonce → caller toàn quyền kiểm soát vị trí nonce trong chuỗi
- `data` là **string** → nonce tự động nối vào cuối với dấu `:` làm separator
- Return: `{ hash, nonce, proof }` — `proof` là giá trị field thật sự được lưu vào graph

Khi `opt.pow` được set, `hash()` gọi đệ quy chính nó với `subOpt` (opt minus pow) để tính hash của mỗi attempt. Nghĩa là `opt.name`, `opt.encode`, `pair`/salt đều được giữ nguyên qua các lần thử.

---

## Nonce base62 thay vì integer

Ban đầu nonce là integer (`0, 1, 2, ...`). Đổi sang **base62** (`0-9A-Za-z`):

- Nonce `0` → `"0"`, nonce `61` → `"z"`, nonce `62` → `"10"`, nonce `3843` → `"zz"`
- Ngắn hơn đáng kể khi search space lớn
- Đẹp hơn trong key: `"order:1712:G"` thay vì `"order:1712:16"`

`intToB62()` được viết trong `hash.js` reuse `base62.ALPHA` từ `src/base62.js` — không duplicate charset string. `ALPHA` được export thêm từ `base62.js` để dùng chung.

---

## Separator `:` thay vì `_`

String mode trước: `data + nonce` → `"order_16"` (nonce liền sau data, không rõ ranh giới).

String mode mới: `data + ":" + nonce` → `"order:G"` — dấu `:` làm ranh giới rõ ràng giữa phần cố định và nonce. Trong function mode, caller tự đặt separator bất kỳ — dấu `:` là convention gợi ý, không ép buộc.

## Tại sao opt.salt không thể làm nonce cho pen PoW?

Câu hỏi hợp lý: `hash()` có sẵn `opt.salt` — sao không dùng nó làm nonce thay vì nhúng nonce vào data?

Trả lời khi trace code:

```js
// Khi opt.name = "SHA-256", code đi qua nhánh ishash():
if (ishash(opt.name)) {
  let hashed = shim.Buffer.from(await digest(data, opt.name), "binary");
  // ...
  return hashed;
}

// digest() gọi thẳng sha256():
async function sha256(data, algorithm) {
  return shim.subtle.digest({ name: algorithm }, new TextEncoder().encode(data));
}
```

`opt.salt` **không được đọc** ở nhánh này. Nó chỉ tồn tại trong nhánh PBKDF2. Thay đổi `opt.salt` không ảnh hưởng gì đến SHA-256 output.

Còn pen.js verifier dùng `{ name: "SHA-256", encode: "hex" }` — pure SHA-256 không có salt. Nếu mining dùng PBKDF2 với opt.salt là nonce, kết quả sẽ hoàn toàn khác với cách pen verify → không tương thích.

Kết luận: **nonce phải nằm trong data (proof)**, không phải trong salt. Đây là lý do kiến trúc, không phải convention.

## Pen PoW compatibility — kiểm chứng

Pen verifier (pen.js dòng 694–709):

```js
runtime.hash(field, null, cb, { name: "SHA-256", encode: "hex" });
// field = regs[pow.field] — giá trị raw của key/val/soul/...
// Chấp nhận nếu hash bắt đầu bằng pow.unit.repeat(pow.difficulty)
```

Mining flow:
1. `proof = data(nonce)` — chuỗi đầy đủ, ví dụ `"order:1712000000:G"`
2. `hash(proof, null, null, { name: "SHA-256", encode: "hex" })` → kiểm tra prefix
3. Khi thắng, `proof` là giá trị thật được write vào graph dưới dạng key
4. Pen verify: `SHA-256(key)` = `SHA-256(proof)` = hash đã tính → pass

Cùng input, cùng algorithm, cùng encoding → cùng output → tương thích tự nhiên.

Test `"hash() mine: proof hashes to same result independently (pen PoW compatible)"` kiểm chứng điều này: mine một lần, hash proof độc lập một lần, so sánh kết quả bằng nhau.

## Breaking change: Compressed pub key (88 → 45 base62 chars)

### Vấn đề

Zen pair format hiện tại lưu public key dưới dạng **uncompressed**: ghép hai tọa độ x và y của điểm EC, mỗi tọa độ 44 chars base62:

```
pub  = base62(x) + base62(y)  →  88 chars
epub = base62(x) + base62(y)  →  88 chars
```

Điều này lãng phí: trên mọi short Weierstrass curve (secp256k1, P-256), biết x và parity của y là đủ để recover y hoàn toàn. Bitcoin và TLS đã dùng compressed format 33 bytes (1 byte prefix `02`/`03` + 32 bytes x) từ đầu.

### Phát hiện trong buổi thảo luận

- Ban đầu tưởng zen dùng base64url (legacy GUN) — nhưng đọc `src/curves/utils.js` thì thấy zen đã dùng base62 từ đầu. `decodeBase64Url` chỉ tồn tại để đọc key cũ từ GUN gốc (backward compat).
- `scalarToString()` và `pointToPub()` trong utils.js đều dùng `base62.biToB62()` — đây là format hiện tại và đúng của zen.
- Format `"evm"` và `"btc"` đã hoạt động đúng — chúng lấy raw BigInt `{x, y}` rồi encode riêng, không qua `pointToPub()`.

### Giải pháp: compressed pub = 44 + 1 char = 45 chars

```
pub  = base62(x) + ("0" | "1")  →  45 chars  (44 chars x + 1 char parity)
epub = base62(x) + ("0" | "1")  →  45 chars
```

Parity char: `"0"` nếu y chẵn, `"1"` nếu y lẻ — tương đương prefix `02`/`03` của compressed pubkey trong Bitcoin.

Recovery y từ x + parity (áp dụng được cho cả secp256k1 lẫn P-256 vì cả hai đều có $p \equiv 3 \pmod{4}$):

$$y = (x^3 + ax + b)^{(p+1)/4} \pmod{p}$$

Nếu parity của y tính được không khớp với parity đã lưu, dùng $p - y$.

### Những gì đạt được

| Chỉ số | Trước | Sau | Cải thiện |
|---|---|---|---|
| pub/epub size | 88 chars | 45 chars | **−49%** |
| Bytes on wire (UTF-8) | 88 B | 45 B | **−43 B mỗi key** |
| Soul shard depth | `ceil(88/2) = 44` | `ceil(45/2) = 23` | **−21 levels** |
| Đọc được compressed pubkey chuẩn | Không | Có thể bridge | Interop tốt hơn |
| Tính nhất quán với Bitcoin/TLS | Không | Khái niệm tương đồng | Dễ explain hơn |

**Bandwidth**: Một sync message chứa 2 keys (`pub` + `epub`) tiết kiệm 86 bytes. Với P2P mesh nhiều peers, savings nhân lên theo số lượng message.

**Storage**: Mỗi user node trong graph giảm ~86 bytes chỉ riêng phần key. Ở scale 1M users, tiết kiệm ~82 MB trước khi tính data khác.

**Shard soul**: Soul của user node dạng `~/XX/XX/.../XX` với 44 segment (88 chars, cắt mỗi 2) → giờ còn 23 segment. Traversal graph nhanh hơn. Index shallower.

### Files bị ảnh hưởng

Chỉ **3 file nguồn** cần sửa:

1. **`src/curves/utils.js`** — thêm `liftX()` (recover y từ x + parity), sửa `pointToPub()`, sửa `parsePub()` (đọc 45-char mới, giữ 88-char cũ để backward compat)
2. **`src/security.js`** — 3 chỗ hardcode `88` → `45`
3. **`test/zen/crypto.js`** — 3 regex/label `{88}` → `{45}`

Các file khác (`format.js`, `pair.js`, `sign.js`, `verify.js`, `secret.js`, `pen.js`) **không cần sửa** vì chúng gọi `parsePub()`/`pointToPub()` qua interface — thay đổi được hấp thụ tại tầng utils.

### Breaking change và backward compat

Đây là **breaking change** đối với dữ liệu đã lưu trên mạng:
- Soul cũ: `~/XX/XX/.../XX` (44 segments từ pub 88 chars)
- Soul mới: `~/XX/XX/.../XX` (23 segments từ pub 45 chars)

**Chiến lược**: `parsePub()` sẽ vẫn đọc được pub 88-char cũ (backward compat đọc). Nhưng mọi key mới sinh ra đều là 45-char. Không có tự động migration soul cũ.

Vì zen là fresh fork, không có deployed data cần migrate — chấp nhận clean break.

## Chain interoperability — phân tích kiến trúc

### Vấn đề cốt lõi

Việc cùng secp256k1 scalar **chưa đủ** để "thông" với ETH/BTC. Cần phân biệt 3 tầng:

```
Tầng 1 — Key identity:   cùng scalar secp256k1  ✅ đã có
Tầng 2 — Address format: mỗi chain encode pub key khác nhau  ✅ format.js đã có
Tầng 3 — Signing:        mỗi chain hash và serialize khác nhau  ❌ chưa có
```

### Tầng 3 — Signing là nơi bị thiếu

**Zen hiện tại:**
```
sign(data) → SHA-256(normalizeMessage(data)) → ECDSA(r, s) → base62-like wrapped { m, s }
```

**ETH cần:**
```
signTx(tx)    → keccak256(rlp(tx)) → recoverable ECDSA(v, r, s) → hex
personalSign  → keccak256("\x19Ethereum Signed Message:\n" + len + msg) → (v, r, s)
```

**BTC cần:**
```
signTx(tx)    → SHA256d(scriptCode) → ECDSA(r, s) DER-encoded → hex
```

Hai điểm khác biệt quan trọng:

1. **Hash function**: ETH dùng keccak256 (đã có trong zen). BTC dùng SHA-256d (double SHA-256, cần thêm).
2. **Recoverable signature**: ETH cần `v` (recovery bit = 0 hoặc 1) để recover public key từ chữ ký mà không cần pub key. Zen hiện tại bỏ qua `v`.

### Recovery bit `v` — cách tính

Trong ECDSA, từ `(r, s)` có thể tồn tại 2 điểm `R` trên curve có cùng `x = r mod N`. `v` cho biết điểm nào là đúng:

```
k*G = R = (r, y)
v = (y & 1n)           // parity của y tọa độ của R (không phải của pub key)
   + (r < N ? 0 : 2)   // nếu x >= N thì thêm 2 (hiếm gặp với secp256k1)
```

Zen đã tính `pt = pointMultiply(k, G)` trong vòng lặp sign — chỉ cần extract `pt.y & 1n` trước khi bỏ qua `pt`.

### Kiến trúc đề xuất

Không thay đổi zen sign/verify hiện tại (backward compat). Thêm 1 module mới `src/ethsign.js` và `src/btcsign.js` hoặc gộp vào `src/chainsign.js`:

```js
// sign ETH personal message
ZEN.sign(msg, pair, null, { hash: "keccak256", prefix: "eth", recoverable: true })
// → { v: 28, r: "0x...", s: "0x..." }  // EIP-191

// sign raw tx hash (pre-hashed by caller)
ZEN.sign(txHash, pair, null, { hash: "raw", recoverable: true })
// → { v, r, s }

// sign BTC tx (SIGHASH_ALL)
ZEN.sign(txBytes, pair, null, { hash: "sha256d", encode: "der" })
// → DER-encoded signature bytes
```

`opt.hash = "raw"` → bỏ qua hash, dùng input trực tiếp làm message hash (32 bytes). Quan trọng vì caller (ethers.js, web3) thường đã hash trước.

`opt.recoverable = true` → output thêm `v`.

`opt.prefix = "eth"` → prepend `"\x19Ethereum Signed Message:\n{len}"` trước khi hash.

`opt.encode = "der"` → DER-encode `(r, s)` thay vì raw 64 bytes (BTC cần DER).

### Điều gì đạt được

Với thiết kế này, một zen pair secp256k1 có thể:
- Sign/verify dữ liệu trong zen graph (SHA-256, format hiện tại)
- Sign ETH personal message và verify với `ecrecover`
- Import `priv` vào MetaMask và có cùng address như `format("evm")`
- Sign BTC transaction (nếu cần)

Tất cả từ **cùng 1 scalar** — không cần quản lý nhiều key.
