## 1. Mining cho pen PoW — `opt.pow` trong `ZEN.hash()`

Việc đầu tiên là thêm khả năng **đào nonce** vào `ZEN.hash()` để hỗ trợ pen policy. Pen có thể yêu cầu một key phải có SHA-256 bắt đầu bằng một prefix nhất định (proof-of-work), và trước đây không có API nào để tìm key hợp lệ — phải tự viết loop bên ngoài.

API mới:

```js
const result = await ZEN.hash(
  nonce => `order:${candle}:${nonce}`,
  null, null,
  { name: "SHA-256", encode: "hex", pow: { unit: "0", difficulty: 2 } }
);
// result: { hash: "00a3f...", nonce: "G", proof: "order:1712000000:G" }
```

`data` có thể là function nhận nonce — caller toàn quyền đặt nonce ở bất kỳ vị trí nào trong chuỗi. Nếu `data` là string thì nonce tự động nối vào cuối với dấu `:` làm separator. Khi `opt.pow` được set, `hash()` gọi đệ quy chính nó với `subOpt` (opt trừ pow) để tính hash của mỗi attempt, giữ nguyên `opt.name`, `opt.encode`, và salt.

**Nonce dùng base62 thay vì integer thuần**: `0, 1, ..., z, 10, 11, ...` — ngắn hơn khi search space lớn, đẹp hơn trong key (`"order:1712:G"` thay vì `"order:1712:16"`). `intToB62()` trong `hash.js` reuse `base62.ALPHA` từ `src/base62.js` để không duplicate charset string.

**Dấu `:` làm separator** được chọn có chủ ý: `"order:G"` rõ ranh giới giữa phần cố định và nonce, không bị nhòa như `"orderG"` trước đây.

**Câu hỏi về `opt.salt`**: Sao không dùng `opt.salt` làm nonce thay vì nhúng vào data? Trace code cho thấy `opt.salt` hoàn toàn không được đọc ở nhánh SHA-256 (`ishash()`) — nó chỉ tồn tại trong nhánh PBKDF2. Pen verifier dùng `{ name: "SHA-256", encode: "hex" }` nên nếu mining dùng PBKDF2 với salt là nonce, output sẽ không khớp với cách pen verify. **Nonce phải nằm trong data (proof)** — đây là lý do kiến trúc, không phải convention.

Pen verifier (pen.js) xác nhận: `runtime.hash(field, null, cb, { name: "SHA-256", encode: "hex" })`. `proof` là giá trị field thật sự được lưu vào graph. Mining flow: `SHA-256(proof)` → kiểm tra prefix → khi thắng, write `proof` vào graph làm key → pen verify `SHA-256(key)` → pass. Cùng input, cùng algorithm, cùng encoding → tương thích tự nhiên.

Thêm 6 test: function data, string data, nonce giữa key, difficulty 2, pen PoW compatibility, callback form. Tất cả pass.

## 2. Compressed pub key — 88 → 45 chars

Từ mining nhảy sang format của public key. Zen lúc đó lưu pub dưới dạng uncompressed: ghép x và y của điểm EC, mỗi tọa độ 44 chars base62 → tổng 88 chars. Lãng phí, bởi vì trên mọi short Weierstrass curve, biết x và parity của y là đủ để recover y hoàn toàn — Bitcoin và TLS đã làm vậy từ đầu.

Trước khi implement có một nhầm lẫn cần làm rõ: ban đầu tưởng zen đang dùng base64url từ GUN legacy. Đọc `src/curves/utils.js` thì thấy zen đã dùng base62 từ đầu — `decodeBase64Url` chỉ tồn tại để đọc key cũ từ GUN gốc. Đây là điểm gây nhầm lẫn cần ghi nhớ vĩnh viễn (xem note đầu trang).

**Compressed format mới**:

```
pub  = base62(x) + ("0" | "1")  →  45 chars  (44 chars x + 1 char parity)
epub = base62(x) + ("0" | "1")  →  45 chars
```

Parity `"0"` nếu y chẵn, `"1"` nếu y lẻ. Recovery y từ x + parity:

$$y = (x^3 + ax + b)^{(p+1)/4} \pmod{p}$$

Công thức đúng cho cả secp256k1 lẫn P-256 vì cả hai đều có $p \equiv 3 \pmod{4}$. Nếu parity không khớp thì lấy $p - y$.

| Chỉ số | Trước | Sau | Cải thiện |
|---|---|---|---|
| pub/epub size | 88 chars | 45 chars | −49% |
| Bytes on wire | 88 B | 45 B | −43 B mỗi key |
| Soul shard depth | 44 segments | 23 segments | −21 levels |
| Scale 1M users | — | tiết kiệm ~82 MB | chỉ riêng phần key |

Thay đổi chỉ cần sửa 3 file nguồn: `src/curves/utils.js` (thêm `liftX`, sửa `pointToPub`, sửa `parsePub` với backward compat cho 88-char cũ), `src/security.js` (3 chỗ hardcode `88` → `45`), `test/zen/crypto.js` (3 regex). Các file khác (`pair.js`, `sign.js`, `verify.js`) gọi qua interface nên không bị ảnh hưởng. **400 tests pass**.

Đây là breaking change đối với soul cũ (44 segments → 23 segments). Vì zen là fresh fork không có deployed data cần migrate — chấp nhận clean break.

## 3. zen.recover — recoverable signature

Compressed pubkey mở ra một câu hỏi tự nhiên: nếu zen đã biết cách recover y từ x, tại sao không recover cả public key từ signature? Đây là nền tảng của `ecrecover` trong Ethereum.

Phân tích kiến trúc interop với chuỗi cần 3 tầng:

```
Tầng 1 — Key identity:   cùng scalar secp256k1  ✅ đã có
Tầng 2 — Address format: mỗi chain encode pub key khác nhau  ✅ format.js đã có
Tầng 3 — Signing:        mỗi chain hash và serialize khác nhau  ❌ chưa có
```

Tầng 3 chưa đủ để thực sự interop với ETH hay BTC — đó là việc của session sau. Nhưng recovery bit `v` có ích ngay lập tức: với `v`, bất kỳ ai nhận được một zen signature cũng có thể biết ai đã ký mà không cần biết pub key trước.

**Implement theo TDD**: viết test trước, code sau. 22 test case bao phủ cơ bản (trả về đúng pub, 45-char, deterministic), cross-check với verify, JS types (null/true/false/0/1/string/object/array), error cases (thiếu `v`, sig tampered), P-256, ZEN instance, cross-curve.

**4 chỗ thay đổi**:

`src/curves/utils.js` — thêm `recoverPub(v, r, s, hashBytes)`. Toán học ECDSA recovery: $R = \text{liftX}(r, v\&1)$, sau đó $Q = u_1G + u_2R$ với $u_1 = -z \cdot r^{-1} \bmod N$ và $u_2 = s \cdot r^{-1} \bmod N$. Tất cả operations dùng A, B, G, N, P từ closure nên tự động đúng cho từng curve.

`src/sign.js` — thêm `v` vào output. Quan trọng: `v` phải tính **SAU** bước low-s normalization vì khi `s > HALF_N` thì `s = N - s` đồng thời `v ^= 1`. Tính trước bước này sẽ ra `v` sai.

`src/recover.js` — module mới, đọc `msg.v` và `msg.c`, khởi tạo đúng curve instance qua `crv(msg.c || opt.curve)`, gọi `recoverPub`, encode bằng `pointToPub`.

`src/index.js` — expose `ZEN.recover` (static) và `zen.recover` (instance).

**Phát hiện quan trọng**: cross-curve recovery không throw. `liftX` luôn tìm được điểm hợp lệ trên curve đích. Kết quả là key sai, không phải lỗi — bảo mật dựa vào field `c` trong sig, không phải vào throw. Test dùng `notStrictEqual` thay vì `rejects`.

Sau rebuild: **422 tests pass, 10 pending**.

## 4. base64 vs base62 cho signature — câu trả lời từ benchmark

Câu hỏi: nên đổi field `s` trong signature từ base64 sang base62 không? Benchmark thực tế (50.000 lần mỗi):

```
base64: 0.17 µs/op  → 88 chars
base62: 7.00 µs/op  → 88 chars
base62 chậm hơn 41x
```

Hai encoding cho ra **cùng độ dài** (88 chars cho 64 bytes). base64 là C native trong Node.js; base62 dùng BigInt division. Không có lý do nào để đổi: không ngắn hơn, chậm hơn 41 lần.

base62 xứng đáng cho scalar (pub, priv, epub) vì những giá trị đó làm soul path, cần ngắn nhất có thể, và chỉ encode/decode một lần khi pair được tạo hoặc load. Signature encode/decode trong mọi lần sign và verify — giữ base64 là quyết định đúng.

## 5. Chain interop — phân tích kiến trúc (chưa implement)

Tầng 3 (signing) cần:

**ETH**: `keccak256("\x19Ethereum Signed Message:\n" + len + msg)` → recoverable `(v, r, s)` hex với `v = 27/28` (EIP-155).

**BTC**: `SHA-256d(scriptCode)` → DER-encoded `(r, s)`.

Thiết kế đề xuất — thêm opt vào `sign()` hiện tại thay vì tạo module riêng:

```js
// ETH personal sign
ZEN.sign(msg, pair, null, { hash: "keccak256", prefix: "eth", recoverable: true })
// → { v: 28, r: "0x...", s: "0x..." }

// ETH raw tx hash (pre-hashed)
ZEN.sign(txHash, pair, null, { hash: "raw", recoverable: true })

// BTC
ZEN.sign(txBytes, pair, null, { hash: "sha256d", encode: "der" })
```

`opt.hash = "raw"` → dùng input trực tiếp làm message hash (caller đã hash trước). `opt.prefix = "eth"` → prepend EIP-191 prefix. `opt.encode = "der"` → DER-encode `(r, s)`. Chưa implement trong session này.

## 6. Bench cleanup — TypeError flood và mislabeled suite

### 6a. bench/06 TypeError flood

`bench/06_crypto_sign.js` báo loạt lỗi mỗi lần chạy. Root cause: gọi `ZEN.pair` sai signature:

```js
// Sai — truyền options object vào vị trí callback
await ZEN.pair({ curve: "secp256k1" });

// Đúng — ZEN.pair(cb, opt), cb đứng trước
await ZEN.pair(null, { curve: "secp256k1" });
```

Khi truyền object làm cb, code cố gọi `object()` → TypeError logged mỗi iteration. Hệ quả phụ: `P2_PAIR` tưởng là P-256 nhưng thực ra lại là secp256k1 (vì `opt.curve` không được đọc) — bench "sign short msg (p256)" đang đo secp256k1. Sau fix, kết quả bench sạch và đúng:

```
pair() secp256k1   7.36 ops/s  (136 ms/op)
pair() p256       10.20 ops/s   (98 ms/op)
sign (k1)         15.24 ops/s   (66 ms/op)
sign (p256)       19.82 ops/s   (50 ms/op)
```

### 6b. bench/01 ZEN.hash "full pipeline" sai tên

Suite `"ZEN.hash (full pipeline)"` gọi `ZEN.hash("hello world")` không có `opt.name`. Kết quả: 36 ops/s (27ms/op). Nhìn vào thì tưởng hash chậm — nhưng không phải. Không có `opt.name` → code đi vào nhánh **PBKDF2 với 100k iterations** — key derivation function, chậm theo thiết kế. Bench chỉ đang label sai.

Fix: tách thành 2 suite rõ ràng:

```
ZEN.hash SHA-256 path:
  short string  40.48K ops/s  (24.7 µs/op)   ← gọi với { name: "SHA-256" }

ZEN.hash PBKDF2 (default, 100k iter):
  short string     32.43 ops/s  (30.8 ms/op)  ← không có opt.name, đúng là chậm
```

Đồng thời giảm `iters` từ 2000 → 100: với PBKDF2, 2000 iters × 27ms = 54 giây chỉ cho suite đó. 100 iters là đủ để có số đáng tin.

### 6c. Bỏ JSON dump khỏi harness

`harness.js` in toàn bộ kết quả bench dưới dạng JSON sau mỗi lần chạy — không ai cần. Bỏ 2 dòng `console.log` đó.

## 7. PBKDF2, HKDF, và lý do akao dùng sai tool

### PBKDF2 có dùng WebCrypto không? Có cách nào nhanh hơn không?

**Có** — `shim.subtle.deriveBits({ name: "PBKDF2" })` là native WebCrypto. Đã chạy ở tốc độ tối đa của hardware rồi. Không thể nhanh hơn nếu vẫn dùng PBKDF2.

Vấn đề thực sự không phải là tốc độ mà là **sai tool**:

- **PBKDF2**: dùng cho **password** (low-entropy input). 100k iterations là để stretch entropy của password lên — cố tình chậm để chống brute-force. Nếu input đã mạnh, iterations là lãng phí hoàn toàn.
- **HKDF**: dùng cho **strong key material** (high-entropy input). Single-pass HMAC, không có iterations, cùng WebCrypto native.

Akao gọi `zen.hash(rawSeed, "wallet")` với `rawSeed` là seed từ WebAuthn — **high-entropy** key material, không phải password. Dùng PBKDF2 ở đây là nhầm tool.

### So sánh thực tế (bench):

```
ZEN.hash SHA-256 path:                40K ops/s   (25 µs/op)   ← bỏ qua salt, không dùng cho key deriv
ZEN.hash HKDF (strong seed, no iter): 6K ops/s   (150 µs/op)  ← đúng tool cho WebAuthn seed
ZEN.hash PBKDF2 (default, 100k iter): 32 ops/s  (30ms/op)     ← đúng cho password
```

HKDF nhanh hơn PBKDF2 **~200 lần**.

**SHA-256 có dùng được cho seed derivation không?** Không — nhánh SHA-256 trong `hash.js` bỏ qua `salt` hoàn toàn. `zen.hash(seed, "wallet", null, { name: "SHA-256" })` và `zen.hash(seed, "avatar", null, { name: "SHA-256" })` trả về **cùng một kết quả** — không phân biệt namespace. Sai hoàn toàn cho key derivation.

| Method | Dùng salt? | Dùng cho seed derivation? | Speed |
|--------|-----------|--------------------------|-------|
| SHA-256 | ❌ bỏ qua | ❌ không | ~40K ops/s |
| HKDF | ✅ | ✅ đúng | ~6K ops/s |
| PBKDF2 | ✅ | ✅ nhưng 200x chậm | ~32 ops/s |

### Nhìn lại GUN work.js (tiền nhiệm của hash.js)

GUN gốc có `SEA.work()` — cùng logic: nhánh `sha*` (bỏ qua salt), còn lại PBKDF2. Không có HKDF. GUN dùng `work()` để hash **password → encryption key** — đúng context cho PBKDF2. Akao kế thừa pattern này nhưng dùng cho context khác (seed derivation) — đó là nguồn gốc của sự chậm.

### Những gì đã thay đổi

**`src/hash.js`** — thêm nhánh HKDF mới trước PBKDF2. `opt.info` là field đặc trưng của HKDF — context string phân biệt các derived key từ cùng seed. Trong call hiện tại, `salt` nhận namespace string (`"wallet"`, `"avatar"`) nên đã đi đúng đường.

**Rebuild `zen.js` và `zen.min.js`**.

**5 file akao** — tất cả seed derivation calls cập nhật sang HKDF:

```js
// wallets/logic.js

// user/logic.js
zen.hash(rawSeed, "avatar", null, { name: "HKDF" })

// avatars/logic.js

// identicons/index.js
```

## Nhìn lại ngày hôm nay

Có một sợi chỉ xuyên qua tất cả: **bỏ đi thứ không cần thiết**. Compressed pubkey bỏ y vì x + parity là đủ. `zen.recover` bỏ yêu cầu truyền pub vào vì signature + v là đủ. HKDF bỏ 100k iterations vì seed đã đủ mạnh — không cần stretch. Mining dùng nonce base62 vì ngắn hơn integer thuần.

Phần còn lại cho ngày sau: chain interop tầng 3 (ETH keccak256 + EIP-191, BTC SHA-256d + DER). Tất cả nền tảng đã có — `v` đã trong sig, `liftX` đã hoạt động, `recoverPub` đã đúng. Chỉ cần wire thêm `opt.hash`, `opt.prefix`, `opt.encode` vào `sign.js`.
