## 1. Phát hiện ban đầu: cascade restart là trigger

Quan sát log từ 13:27 (hôm trước): tất cả 3 VPS restart **đồng thời** do `zen-update.timer`
(OnUnitActiveSec=1h, không có jitter). Trong ~20 giây, cả 3 node cùng offline → khi online lại,
bão kết nối flood nhau, gây mất trạng thái peer.

**Fix ngay lập tức:** Thêm `RandomizedDelaySec=10min` vào `/etc/systemd/system/zen-update.timer`
trên cả 3 node để tránh cascade restart trong tương lai.

## 2. Bug A — BOOT-WATCHDOG chỉ xóa PEX tombstone, bỏ sót WebSocket tombstone

### Triệu chứng

Sau cascade restart, BOOT-WATCHDOG log "Reconnecting..." liên tục mỗi 30s cho zen1, nhưng kết
nối không bao giờ được thiết lập lại.

### Root cause

Hệ thống có **hai tombstone riêng biệt**:
- `opt._tombUrls` — PEX tombstone (chặn PEX adopt)
- `opt.tomb` — WebSocket tombstone (chặn `open()` khởi tạo kết nối)

BOOT-WATCHDOG cũ chỉ xóa `opt._tombUrls`. Sau cascade restart, `websocket.js onclose` thêm
URL vào `opt.tomb` (qua `_axeGuess >= 5` hoặc `_hiGuess >= 3`). WATCHDOG không clear `opt.tomb`
→ `open()` kiểm tra `opt.tomb` trước tiên và return sớm → không bao giờ reconnect.

```js
// Cũ — thiếu
opt._tombUrls.delete(norm);

// Mới — xóa cả hai
opt._tombUrls.delete(norm);
opt.tomb.delete(norm);
opt.tomb.delete(PeerRegistry.alt(norm));  // xóa cả dạng wss://
```

## 3. Bug B — `_isBoot=true` bị mất do race condition trong BOOT-WATCHDOG

### Root cause

BOOT-WATCHDOG cũ tạo **object peer mới** mỗi lần reconnect:

```js
// Cũ — race condition
const watchPeer = { id: norm, url: norm };
watchPeer._isBoot = true;
route.hi(watchPeer);  // ← open() xong, wire.onopen fire
// wire.onopen gọi mesh.hi(peer) → opt.peers[url] = peer (object MỚI, không có _isBoot!)
```

Khi `wire.onopen` fire, `mesh.hi()` lưu lại object peer mới (không có `_isBoot=true`) vào
`opt.peers[url]`. Sau đó khi kết nối đóng, `websocket.js` thấy `!peer._isBoot` → tăng
`_axeGuess`/`_hiGuess` → tombstone → loop vô hạn.

```js
// Mới — dùng lại object hiện tại
const watchPeer = (opt.peers && opt.peers[norm]) || { id: norm, url: norm };
watchPeer._isBoot = true;   // set TRƯỚC khi gọi route.hi()
delete watchPeer._noReconnect;
delete watchPeer._hiGuess;
delete watchPeer._axeGuess;
route.hi(watchPeer);
// wire.onopen: mesh.hi lưu lại object NÀY (đã có _isBoot=true) ✓
```

## 4. Bug C — `dam:"pex"` handler làm hỏng `_pidx`, khiến BOOT entry bị evict

### Đây là bug quan trọng nhất và tinh vi nhất.

### Triệu chứng

Sau khi fix A và B, zen0 vẫn BOOT-WATCHDOG cho zen1. Debug log cho thấy `entry.pub` của zen1
mãi là `undefined` — entry bị evict khỏi registry rồi được tạo lại (nopub) sau mỗi 30s.

### Root cause

`dam:"pex"` handler trong `server.js` gọi:
```js
registry.confirm(gossipedUrl, { pub: peer.pub })
```

cho **mọi URL** trong danh sách gossip, dùng pub của **peer gửi tin** (inbound peer).

PEX re-gossip của relay gửi single-URL messages (`peers: ["wss://zen1.akao.io:8420/zen"]`) cho
mỗi peer đã confirm. Giả sử zen1 đang nhận re-gossip từ zen, message này tới zen0:

```
peer (inbound) = zen  (pub = 0NRcZMmN...)
msg.peers = ["wss://zen1.akao.io:8420/zen"]
```

Handler gọi: `registry.confirm("zen1.url", { pub: "0NRcZMmN..." })`

→ preg.js `_pidx`: `{ "0NRcZMmN..." → zen1.url }` ← **SAI!**

Sau đó, khi xác nhận outbound của zen thực sự đến (`dam:"?"` outbound):
`registry.confirm("zen.url", { pub: "0NRcZMmN..." })`

→ preg.js tìm `_pidx["0NRcZMmN..."] = zen1.url ≠ zen.url` → **EVICT zen1.url** như một wildcard alias!

zen1 bị xóa khỏi registry → BOOT-WATCHDOG tạo entry mới nopub → không thể xác nhận liveness →
WATCHDOG fire mỗi 30s → vòng lặp vô hạn.

### Fix

Không bao giờ dùng `peer.pub` cho gossip URLs trong `dam:"pex"`. Thay vào đó, dùng
**DNS-based inbound matching** để map pub → BOOT URL một cách đáng tin cậy.

```js
// Cũ — SAI, gây _pidx corruption
registry.confirm(gossipedUrl, { pub: peer.pub });

// Mới — chỉ update seen/lastOk, không pub
registry.touch(url);
```

## 5. Fix: DNS-based inbound BOOT peer matching

Để có thể map inbound connection → BOOT URL (cần cho `isPeerAlive()` với AXE inbound-only),
thêm cơ chế DNS-based IP lookup lúc startup:

```js
// Startup: resolve tất cả BOOT hostname → IP → Map
const bootIpToUrl = new Map();
for (const entry of registry.bootEntries()) {
  const hostname = new URL(entry.url).hostname;
  const addrs = await dns.lookup(hostname, { all: true, family: 0 }); // cả IPv4 và IPv6
  for (const { address } of addrs) {
    bootIpToUrl.set(address, entry.url);
  }
}

// dam:"?" inbound handler
const srcIp = sock.remoteAddress.replace(/^::ffff:/, "");
const bootUrl = bootIpToUrl.get(srcIp);
if (bootUrl) {
  peer._bootUrl = bootUrl;
  registry.confirm(bootUrl, { pub: peer.pub, pid: peer.pid });
}
```

**Lý do cần thiết:** Sau AXE conflict resolution, BOOT peer thường chỉ tồn tại dưới dạng
INBOUND connection (AXE drop outbound của một bên). `opt.peers[url].wire` là null (outbound
không có). Phải scan `axe.up` theo `._bootUrl` để xác nhận peer còn sống.

**IPv6 trên pi:** `zen.akao.io` có AAAA record `2407:3640:2328:2041::1`. pi (IPv6-only)
resolve cả A lẫn AAAA → `bootIpToUrl` có cả hai → inbound từ IPv6 vẫn match đúng.

## 6. Fix `isPeerAlive()` — nopub fallback cho inbound-only connections

```js
// Cũ — chỉ check outbound wire
const alive2 = !!(opt.peers[url] && opt.peers[url].wire);

// Mới — thêm scan axe.up theo _bootUrl
const alive2 = !!(opt.peers[url] && opt.peers[url].wire);
if (!alive2) {
  const axeUp = getAxeUp();
  const byInboundUrl = Object.values(axeUp).some(ap => ap && ap.wire && ap._bootUrl === url);
  if (byInboundUrl) return true;
}
```

## 7. Bug D — DEAD(nopub) spam cho PEX-gossiped third-party peers

### Triệu chứng

Sau khi deploy 3 fix trên, vẫn còn `[BOOT-WATCHDOG] DEAD(nopub): http://20.55.86.49` mỗi 30s
trên cả 3 VPS. `20.55.86.49` là Azure IP — relay bên thứ 3 connect inbound qua PEX discovery,
sau đó disconnect.

### Root cause

`dam:"pex"` handler (trước fix D) gọi `registry.confirm(url, {})` — không có pub, nhưng vẫn
set `lastOk = Date.now()`. Entry này xuất hiện trong `confirmedNonBoot()` (lọc theo `lastOk > 0`)
nhưng không có pub/pid → `isPeerAlive()` không thể xác nhận → log `DEAD(nopub)` mỗi 30s.

### Fix

1. `dam:"pex"` handler: dùng `registry.touch(url)` thay vì `confirm({})` — chỉ update `seen`,
   không set `lastOk`, nên entry không xuất hiện trong `confirmedNonBoot()`.

2. Non-BOOT WATCHDOG loop: skip entries không có pub lẫn pid:
   ```js
   if (!entry.pub && !entry.pid) continue; // không thể verify liveness
   ```

## Bài học

**1. Hai tombstone systems phải được xóa cùng nhau.** `opt._tombUrls` và `opt.tomb` độc lập
nhau. Mọi code clear tombstone phải clear cả hai — nếu không, half-clear vẫn block kết nối.

**2. Object identity quan trọng khi có async callbacks.** `wire.onopen` lưu lại object vào
`opt.peers[url]`. Nếu tạo object mới trước `open()`, object cũ (có `_isBoot`) bị mất sau
`onopen`. Luôn dùng lại object hiện có hoặc set flag **sau** `onopen`.

**3. Không bao giờ dùng pub của inbound peer cho gossip URLs.** PEX re-gossip gửi single-URL
messages — không thể phân biệt "peer đang gossip URL của mình" vs "peer đang re-gossip URL
của người khác". Dùng DNS-based IP matching để map pub → URL, không dùng gossip context.

**4. `confirm({})` vs `touch()` có ngữ nghĩa khác nhau.** `confirm()` set `lastOk` → entry
vào `confirmedNonBoot()` → WATCHDOG monitor. `touch()` chỉ update `seen` → không vào WATCHDOG.
Chỉ gọi `confirm()` khi thực sự có kết nối WebSocket thành công.

**5. AXE inbound-only là trạng thái bình thường, không phải lỗi.** Sau conflict resolution,
một bên giữ outbound, bên kia chỉ có inbound. `isPeerAlive()` phải check cả hai hướng.
