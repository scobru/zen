What this browser bundle loads

AXE is built into `zen.min.js`.

Since the AXE peer-mesh migration, AXE is part of the core bundle — no separate import needed. WebRTC, Radisk, Radix, and OPFS remain optional add-on modules. You still need a real /zen relay endpoint to see relay-assisted clustering behavior rather than local-only startup.

            Browser notes
            
The AXE question, answered directly

          

          

            You asked whether AXE is already inside ZEN in the browser. The short answer is still no: AXE is an optional add-on, not part of `zen.min.js`. But it can now be imported in the browser path again; the remaining requirement is a real relay endpoint if you want live AXE peering behavior.
          

        

        

          

            
Is AXE already inside `zen.min.js`?

            

              Yes. AXE peer-mesh is built into the `zen.min.js` core bundle — no separate import needed. Just new ZEN({ peers: [...] }) and AXE is automatically active.
            

          

          

            
Then how do I use AXE in browser code?

            

              Import `zen.min.js` and instantiate: new ZEN({ peers: ["wss://your-relay/zen"] }). AXE clustering is on by default. To see real peering, point peers at an actual relay endpoint that serves /zen.
            

          

          

            
What about WebRTC, Radisk, Radix, and OPFS?

            

              Same model: they are separate browser-side modules that extend or activate runtime behavior. This page imports them all in one browser entrypoint so another project can copy one module and get the same browser stack.
            

          

          

            
Can another project use this page as a starting point?

            

              Yes. The page is intentionally structured around raw `new ZEN(...)`, `get`, `put`, `on`, `pair`, `sign`, `verify`, `recover`, `encrypt`, `decrypt`, and `ZEN.pen(...)` flows rather than hiding them behind framework abstractions.
